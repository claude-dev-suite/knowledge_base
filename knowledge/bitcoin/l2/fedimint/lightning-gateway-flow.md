# Fedimint Lightning Gateway Flow - Deep Dive

> Phase B article. Companion to dev-suite skill `bitcoin/l2/fedimint`.
> Canonical source: https://github.com/fedimint/fedimint/tree/master/gateway
> Skill source: https://github.com/claude-dev-suite/claude-dev-suite/blob/main/skills/bitcoin/l2/fedimint/SKILL.md

## Concept

A **Lightning gateway** bridges a Fedimint federation's Chaumian e-cash to
the public Lightning network. It lets users send/receive Lightning
payments while keeping their funds inside the federation. The gateway
holds a hot Lightning node and acts as a counterparty in HTLC swaps with
the federation.

## Walkthrough / mechanics

### Roles

- **Federation**: holds e-cash, runs HBBFT consensus, controls on-chain
  peg.
- **Gateway**: external entity (often a federation guardian, but can be
  third-party) running an LND/CLN node + the `gatewayd` daemon.

### Send: e-cash -> Lightning invoice

User pays an external BOLT11 invoice using e-cash:

1. User submits BOLT11 + e-cash notes to the federation `pay_invoice`
   endpoint, choosing a gateway.
2. Federation locks the e-cash notes into an **outgoing contract** on the
   federation: `(payment_hash, amount, gateway_pk, expiry)`.
3. Gateway pays the BOLT11 over Lightning using its own node funds.
4. On `payment_preimage` reveal, gateway publishes the preimage to the
   federation contract.
5. Federation atomically transfers e-cash to gateway (claim) and burns
   the user's locked notes.

This is a HTLC analog: the federation contract acts as the on-chain HTLC,
the gateway provides liquidity.

### Receive: Lightning -> e-cash

User receives a Lightning payment as new e-cash:

1. User asks federation: "make me an invoice for `v` sats".
2. Federation collaborates with gateway: gateway constructs a BOLT11
   with `payment_hash = H(preimage)`, where preimage is generated by
   federation. Federation gives only `H(preimage)` to gateway.
3. Gateway publishes invoice to user.
4. External payer pays invoice over Lightning. Gateway holds the inbound
   HTLC (HODL invoice or post-settlement re-locking).
5. Gateway sends preimage to federation, which mints fresh blinded
   e-cash notes worth `v` to user. Gateway claims the inbound HTLC.

### Trust model

- User trusts federation (already by virtue of joining).
- Federation does NOT trust gateway — gateway must escrow correct amount
  in the federation contract before Lightning settlement.
- Gateway's risk: user could double-spend e-cash before federation
  recognises it; mitigated by HBBFT atomic state transitions per round.

### Multi-gateway routing

A federation typically registers many gateways. Client selects by:

- Lower fee.
- Sufficient inbound liquidity (for receive).
- Geographic / latency preference.

If a chosen gateway fails (offline, route-not-found), client retries with
another.

## Worked example

Alice in federation F, sending 50 000 sat to external Bob's BOLT11.

```
1. Alice -> F: pay-invoice request, e-cash notes (50 000 + fee)
2. F locks contract (hash=H1, amount=50 000, gw=G, expiry=t+10min)
3. G pays Bob's BOLT11 over LN -> Bob's node responds with preimage P1
4. G publishes P1 to F contract
5. F transfers 50 000 e-cash to G, burns Alice's notes
```

End state:
- Alice: -50 000 + fee e-cash
- Bob: +50 000 sat on Lightning
- G: net zero (paid LN, gained e-cash)

## Common pitfalls

- **Gateway liquidity mismatch**: user-side amount in fed e-cash must be
  available as outbound LN liquidity at gateway. UX must surface
  failures clearly.
- **Sub-sat amounts**: BOLT11 amount is millisat granular but federation
  e-cash is sat granular; gateway absorbs rounding.
- **Privacy from gateway**: gateway sees `payment_hash` and route hints.
  In send mode, payment_hash links to public LN payment; payer privacy
  is similar to ordinary LN (no extra leak).
- **Receive side requires HODL invoice** (or careful timing): if gateway
  publishes preimage to federation BEFORE LN HTLC settles, attacker
  could double-spend. Modern gatewayd uses HODL with explicit settle on
  preimage publish.
- **Federation downtime**: if HBBFT halts mid-payment, gateway's funds
  may be locked in contract awaiting settlement. Long expiries +
  monitoring required.
