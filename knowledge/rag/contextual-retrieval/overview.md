# Contextual Retrieval

## Overview / TL;DR

Contextual retrieval is a technique published by Anthropic in September 2024 that prepends chunk-specific context to each chunk before embedding. Standard RAG chunks often lose meaning when separated from their parent document -- a chunk saying "the company was founded in 1983" is useless without knowing which company. Contextual retrieval fixes this by using an LLM (Claude Haiku) to generate a brief contextual preamble for each chunk that situates it within the full document. When combined with contextual BM25 and reranking, this technique reduces retrieval failure rates by 67% compared to naive RAG. The technique is conceptually simple but requires careful implementation to control ingestion costs.

---

## The Problem: Chunks Lose Context

Consider a technical document about configuring PostgreSQL replication. After chunking, you might get:

**Chunk 42**: "Set the `wal_level` parameter to `replica` and restart the server. The `max_wal_senders` should be at least 3 for a typical setup."

This chunk is technically correct but lacks context:
- Which database system is this for? (PostgreSQL)
- What feature is being configured? (Streaming replication)
- Which section of the document is this from? (Primary server setup)

A query like "How do I set up PostgreSQL replication?" might not match this chunk well because the chunk never mentions "PostgreSQL" or "replication."

---

## The Solution: Prepend Context

Before embedding, prepend a contextual preamble generated by an LLM that has seen the full document:

**Contextualized Chunk 42**: "This chunk is from the PostgreSQL Administration Guide, specifically the section on configuring streaming replication on the primary server. It describes the WAL (Write-Ahead Logging) configuration parameters needed for replication. Set the `wal_level` parameter to `replica` and restart the server. The `max_wal_senders` should be at least 3 for a typical setup."

Now the chunk contains the keywords "PostgreSQL," "streaming replication," "primary server," and "WAL" -- making it much more retrievable.

---

## How It Works: The Pipeline

```
Original Document
      |
      v
  [Chunk into N pieces]
      |
      v
  For each chunk:
      +---> Send (full document + chunk) to Claude Haiku
      |     with the contextual retrieval prompt
      |
      +---> Receive: short context preamble (50-100 tokens)
      |
      +---> Prepend context to chunk text
      |
      v
  [Embed the contextualized chunk]
      |
      v
  [Store in vector DB + BM25 index]
```

### The Prompt (Anthropic's Published Version)

```python
CONTEXTUAL_RETRIEVAL_PROMPT = """<document>
{WHOLE_DOCUMENT}
</document>
Here is the chunk we want to situate within the whole document:
<chunk>
{CHUNK_CONTENT}
</chunk>
Please give a short succinct context to situate this chunk within
the overall document for the purposes of improving search retrieval
of the chunk. Answer only with the succinct context and nothing else."""
```

### Basic Implementation

```python
import anthropic
from dataclasses import dataclass


@dataclass
class ContextualizedChunk:
    original_text: str
    context: str
    contextualized_text: str
    metadata: dict


client = anthropic.Anthropic()


def generate_chunk_context(
    document_text: str,
    chunk_text: str,
) -> str:
    """Generate contextual preamble for a single chunk."""
    response = client.messages.create(
        model="claude-haiku-4-20250514",
        max_tokens=200,
        temperature=0,
        messages=[{
            "role": "user",
            "content": CONTEXTUAL_RETRIEVAL_PROMPT.format(
                WHOLE_DOCUMENT=document_text,
                CHUNK_CONTENT=chunk_text,
            ),
        }],
    )
    return response.content[0].text.strip()


def contextualize_chunks(
    document_text: str,
    chunks: list[dict],
) -> list[ContextualizedChunk]:
    """Contextualize all chunks from a document."""
    results = []
    for chunk in chunks:
        context = generate_chunk_context(document_text, chunk["text"])
        contextualized = ContextualizedChunk(
            original_text=chunk["text"],
            context=context,
            contextualized_text=f"{context}\n\n{chunk['text']}",
            metadata=chunk.get("metadata", {}),
        )
        results.append(contextualized)
    return results
```

## The Three-Layer Stack

Anthropic's results show that stacking three techniques produces the best results:

### Layer 1: Contextual Embeddings

Embed the contextualized chunk text (context preamble + original chunk). This improves semantic search by adding document-level keywords to the chunk's embedding.

**Impact**: 35% reduction in retrieval failure rate (top-20 retrieval).

### Layer 2: Contextual BM25

Index the contextualized chunk text in a BM25 keyword index. The added context ensures that important keywords from the document title, section heading, and surrounding content are present in the chunk for keyword matching.

**Impact**: Additional 19% reduction in failure rate when combined with contextual embeddings (total: 49%).

### Layer 3: Reranking

Apply a cross-encoder or neural reranker to the combined results from layers 1 and 2 before passing to the LLM.

**Impact**: Additional 18% reduction when stacked on top of layers 1 and 2 (total: 67%).

```python
"""Full three-layer contextual retrieval pipeline."""

import anthropic
from sentence_transformers import SentenceTransformer, CrossEncoder
from rank_bm25 import BM25Okapi
import numpy as np


class ContextualRetrievalPipeline:
    def __init__(self):
        self.client = anthropic.Anthropic()
        self.embed_model = SentenceTransformer("BAAI/bge-base-en-v1.5")
        self.reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")
        self.chunks = []          # Contextualized chunk texts
        self.embeddings = None    # Numpy array of embeddings
        self.bm25 = None          # BM25 index

    # --- Ingestion ---

    def ingest_document(self, document_text: str, chunks: list[dict]):
        """Contextualize, embed, and index all chunks from a document."""
        contextualized = []
        for chunk in chunks:
            context = generate_chunk_context(document_text, chunk["text"])
            ctx_text = f"{context}\n\n{chunk['text']}"
            contextualized.append({
                "text": ctx_text,
                "original": chunk["text"],
                "context": context,
                "metadata": chunk.get("metadata", {}),
            })

        # Store chunks
        self.chunks.extend(contextualized)

        # Build embedding index
        texts = [c["text"] for c in self.chunks]
        self.embeddings = self.embed_model.encode(
            texts, normalize_embeddings=True
        )

        # Build BM25 index on contextualized text
        tokenized = [t.lower().split() for t in texts]
        self.bm25 = BM25Okapi(tokenized)

    # --- Retrieval ---

    def retrieve(
        self,
        query: str,
        top_k: int = 5,
        initial_k: int = 25,
        bm25_weight: float = 0.3,
    ) -> list[dict]:
        """Three-layer retrieval: contextual embeddings + BM25 + reranking."""
        # Layer 1: Contextual embedding search
        query_emb = self.embed_model.encode(query, normalize_embeddings=True)
        vector_scores = np.dot(self.embeddings, query_emb)

        # Layer 2: Contextual BM25 search
        bm25_scores = self.bm25.get_scores(query.lower().split())

        # Normalize both score arrays to [0, 1]
        v_min, v_max = vector_scores.min(), vector_scores.max()
        if v_max > v_min:
            vector_norm = (vector_scores - v_min) / (v_max - v_min)
        else:
            vector_norm = np.zeros_like(vector_scores)

        b_min, b_max = bm25_scores.min(), bm25_scores.max()
        if b_max > b_min:
            bm25_norm = (bm25_scores - b_min) / (b_max - b_min)
        else:
            bm25_norm = np.zeros_like(bm25_scores)

        # Fuse scores
        fused = (1 - bm25_weight) * vector_norm + bm25_weight * bm25_norm
        top_indices = np.argsort(fused)[::-1][:initial_k]

        # Layer 3: Reranking
        candidates = [self.chunks[i] for i in top_indices]
        pairs = [(query, c["text"]) for c in candidates]
        rerank_scores = self.reranker.predict(pairs)

        for i, candidate in enumerate(candidates):
            candidate["score"] = float(rerank_scores[i])

        candidates.sort(key=lambda x: x["score"], reverse=True)
        return candidates[:top_k]
```

The following numbers are from Anthropic's blog post, tested across multiple knowledge domains:

| Configuration | Failure Rate Reduction |
|--------------|----------------------|
| Contextual Embeddings only | 35% |
| Contextual Embeddings + Contextual BM25 | 49% |
| Contextual Embeddings + Contextual BM25 + Reranking | **67%** |

### By Domain

| Domain | Traditional RAG Failure Rate | Contextual RAG + Reranking Failure Rate | Improvement |
|--------|-----------------------------|-----------------------------------------|-------------|
| Codebase | ~25% | ~8% | 68% reduction |
| Science papers | ~35% | ~12% | 66% reduction |
| Fiction novels | ~30% | ~10% | 67% reduction |

### What "Failure Rate" Means

A retrieval failure occurs when the correct chunk (containing the answer) is NOT in the top-20 retrieved results. This is measured against a curated evaluation set where the correct chunk is known. Reducing the failure rate from 25% to 8% means the system goes from failing on 1 in 4 queries to failing on 1 in 12.

## Batch Processing Pipeline

Processing chunks individually is slow and expensive. A production pipeline processes in batches.

```python
import asyncio
import anthropic
from typing import Optional


async def batch_contextualize(
    document_text: str,
    chunks: list[dict],
    concurrency: int = 10,
    model: str = "claude-haiku-4-20250514",
) -> list[dict]:
    """Process chunks in parallel with rate limiting."""
    client = anthropic.AsyncAnthropic()
    semaphore = asyncio.Semaphore(concurrency)

    async def process_one(chunk: dict) -> dict:
        async with semaphore:
            response = await client.messages.create(
                model=model,
                max_tokens=200,
                temperature=0,
                messages=[{
                    "role": "user",
                    "content": CONTEXTUAL_RETRIEVAL_PROMPT.format(
                        WHOLE_DOCUMENT=document_text,
                        CHUNK_CONTENT=chunk["text"],
                    ),
                }],
            )
            context = response.content[0].text.strip()
            return {
                "text": f"{context}\n\n{chunk['text']}",
                "original": chunk["text"],
                "context": context,
                "metadata": chunk.get("metadata", {}),
            }

    tasks = [process_one(chunk) for chunk in chunks]
    return await asyncio.gather(*tasks)
```

## Prompt Caching for Cost Reduction

The full document is repeated in every chunk's prompt. Anthropic's prompt caching (ephemeral cache) reduces this cost by up to 90%.

```python
import anthropic

client = anthropic.Anthropic()


def contextualize_with_caching(
    document_text: str,
    chunks: list[dict],
) -> list[dict]:
    """Use prompt caching to avoid re-sending the full document each time.

    The document is sent once and cached; subsequent calls reference the cache.
    Cached tokens cost 90% less than uncached tokens.
    """
    results = []
    for chunk in chunks:
        response = client.messages.create(
            model="claude-haiku-4-20250514",
            max_tokens=200,
            temperature=0,
            system=[
                {
                    "type": "text",
                    "text": f"<document>\n{document_text}\n</document>",
                    "cache_control": {"type": "ephemeral"},
                }
            ],
            messages=[{
                "role": "user",
                "content": (
                    f"Here is the chunk we want to situate within the whole document:\n"
                    f"<chunk>\n{chunk['text']}\n</chunk>\n"
                    f"Please give a short succinct context to situate this chunk within "
                    f"the overall document for the purposes of improving search retrieval "
                    f"of the chunk. Answer only with the succinct context and nothing else."
                ),
            }],
        )
        context = response.content[0].text.strip()
        results.append({
            "text": f"{context}\n\n{chunk['text']}",
            "original": chunk["text"],
            "context": context,
            "metadata": chunk.get("metadata", {}),
            "cache_creation_tokens": response.usage.cache_creation_input_tokens,
            "cache_read_tokens": response.usage.cache_read_input_tokens,
        })
    return results
```

**How caching works**:

1. The first chunk's request sends the full document. Anthropic caches it (cache write).
2. Subsequent requests for chunks from the same document hit the cache (cache read).
3. Cache reads cost 10% of normal input token pricing.
4. Cache has a 5-minute TTL -- process all chunks from one document before moving to the next.

**Cost comparison** (10,000-token document, 25 chunks, Haiku pricing):

| Approach | Input Tokens | Cost |
|----------|-------------|------|
| No caching | 25 x 10,000 = 250,000 | $0.20 |
| With caching | 10,000 (first) + 24 x 10,000 x 0.1 = 34,000 effective | ~$0.03 |
| Savings | | 85% |

## When to Use Contextual Retrieval

**Strong fit**:
- Chunks frequently lose important keywords (document title, subject, entity names)
- Queries use different vocabulary than the chunks
- Corpus has long documents where chunks are far from section headings
- You need high retrieval accuracy and can tolerate slower ingestion

**Weak fit**:
- Chunks are already self-contained (e.g., FAQ entries, product descriptions)
- Documents are very short (1-2 paragraphs -- no context to lose)
- Ingestion cost is a hard constraint and prompt caching is not available
- Real-time ingestion is required (contextual generation adds latency)

---

## Common Pitfalls

1. **Not using prompt caching.** Without caching, contextual retrieval is 5-10x more expensive than necessary. Always process all chunks from one document in sequence to maximize cache hits.
2. **Generating context that is too long.** The context preamble should be 50-100 tokens. Longer contexts dilute the original chunk's embedding signal. Limit `max_tokens=200` in the API call.
3. **Not including BM25.** Contextual embeddings alone give a 35% improvement. Adding contextual BM25 pushes it to 49%. The BM25 component is cheap (no API cost) and catches keyword matches that embeddings miss.
4. **Skipping reranking.** The reranking layer adds another 18 percentage points. Without it, you leave significant accuracy on the table.
5. **Processing chunks from different documents interleaved.** This breaks prompt caching. Process all chunks from document A, then all chunks from document B.
6. **Using the context for generation, not just retrieval.** The context preamble is optimized for search, not for the final answer. Pass the original chunk text (not the contextualized version) to the generation LLM to avoid confusion from the prepended context.
