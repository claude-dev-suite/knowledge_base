# ISA-88 / ISA-95 - Batch Control and Enterprise Integration

> Official Reference: https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa88
> ISA-95 Reference: https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa95
> Industry Overview: https://www.automation.com/en-us/articles/2013/isa-88-batch-control-standard

## Overview

ISA-88 (ANSI/ISA-88, IEC 61512) is the international standard for batch process control. It defines a **physical model** (equipment hierarchy from Enterprise down to Control Module), a **procedural model** (recipe hierarchy from Procedure down to Phase), and **state machines** that govern how batch operations execute.

ISA-95 (ANSI/ISA-95, IEC 62264) defines the standard integration interface between enterprise systems (ERP, MES) and control systems (DCS, SCADA). Together, ISA-88 + ISA-95 define the complete vertical hierarchy from field sensors to business planning.

## 1. ISA-88 Physical Model

The physical model defines the equipment hierarchy. Each level is an abstraction of physical plant equipment with specific control responsibilities.

```
Enterprise
  └── Site
        └── Area
              └── Process Cell
                    └── Unit
                          └── Equipment Module (EM)
                                └── Control Module (CM)
```

### Level Descriptions

**Enterprise**
The company or corporation. Multiple sites may exist under one enterprise. Not directly relevant to control, but provides organizational context for recipe management and production scheduling (ISA-95 domain).

**Site**
A physical manufacturing facility defined by geographic location. A site may contain multiple production areas using different processes.

**Area**
A functional section of the plant defined by product type, geographical boundaries, or process purpose. Examples: "Powder Preparation Area", "Liquid Blending Area", "Filling and Packaging Area". The Area code maps directly to the ISA-5.1 tag area prefix (e.g., area "113" → tag "11301.FIC.056A").

**Process Cell**
The smallest self-contained collection of equipment that can independently produce a batch. A Process Cell includes all Units, Equipment Modules, and Control Modules required to execute one or more recipes without depending on equipment outside the cell. A typical Process Cell contains 1-10 Units.

**Unit**
The central equipment entity in ISA-88. A Unit is a collection of equipment that performs one major processing step at a time.

Key characteristics:
- A Unit can execute **only one batch operation at a time** — this is a fundamental constraint
- The Unit hosts all Equipment Modules and Control Modules needed for its function
- A Unit has a defined state machine matching the ISA-88 Phase State Machine
- Examples: Reactor R-101 (with agitator, jacket, feeds), Blender B-201, Spray Dryer D-301, Crystallizer C-401

**Equipment Module (EM)**
A collection of equipment that performs a limited, specific minor processing activity. Key characteristics:
- Can be **shared between Units** — this is the key difference from a Unit (which is dedicated to one batch at a time)
- Has its own state machine
- Contains one or more Control Modules
- Requests for shared EMs from multiple Units are managed by arbitration logic
- Examples: Weighing/Dosing System (shared feeder for multiple reactors), Jacket Temperature Circuit, CIP Skid, Solvent Recovery System

**Control Module (CM)**
The lowest addressable level of equipment-oriented control. A CM directly interfaces with field I/O and performs one specific control function.

Key characteristics:
- Executes **continuously** (every scan cycle) — not sequence-based
- Directly drives outputs to field devices and reads inputs from sensors
- Has Modes: AUTO (commands from sequence/EM), MANUAL (operator direct), LOCKED/OUT-OF-SERVICE
- Examples: Single motor with start/stop/feedback, Single valve with open/close/position, Single PID control loop, Single transmitter with scaling and alarms

## 2. ISA-88 Procedural Model

The procedural model defines the execution hierarchy: **what** to do, organized as a hierarchy of callable units.

```
Recipe
  └── Procedure
        └── Unit Procedure
              └── Operation
                    └── Phase
```

### Level Descriptions

**Recipe**
The complete set of information required to produce a specific product batch. Consists of: Header, Formula (ingredients/quantities), Equipment Requirements, and the Procedure definition. See section 4 for recipe types.

**Procedure**
The top-level procedural element. Coordinates all Unit Procedures to produce one complete batch. For a single-unit process, there is typically one Procedure with one Unit Procedure. For multi-unit processes (e.g., reactor → crystallizer → dryer), the Procedure coordinates all units.

**Unit Procedure**
A sequence of Operations executed in one specific Unit from batch start to finish. The Unit Procedure is the bridge between the equipment hierarchy (Unit) and the procedure hierarchy. One Unit Procedure runs in one Unit; a batch may have multiple Unit Procedures running concurrently on different Units.

**Operation**
An ordered set of Phases that produces a chemically or physically defined change to the material. Operations are the "chapters" of the processing story, each with a clear beginning and end state.

Examples for a batch reactor:
- Charge Operation: fill the reactor with feed materials
- React Operation: heat, hold at temperature, monitor reaction
- Discharge Operation: transfer product out, clean

**Phase**
The smallest element of the procedural model. A Phase is:
- Self-contained and independently executable
- Associated with one Unit (or shared EM via Unit)
- Governed by the ISA-88 Phase State Machine
- Has defined input Parameters (provided by recipe before START) and Output Parameters (results reported after COMPLETE)
- The level at which DCS implementation occurs

Phase examples for a reactor batch:
- `Fill_Solvent`: opens inlet valve V-101, monitors level LT-101, closes valve when target volume reached
- `HeatUp`: ramps jacket temperature from ambient to 80°C at 2°C/min
- `HoldReact`: maintains 80°C for 120 minutes (recipe parameter), monitors conversion
- `Cool`: cool jacket to 30°C, verify product temperature before proceeding
- `Transfer`: opens outlet valve, starts transfer pump, monitors destination tank level

### Procedure Hierarchy Example

```
Recipe: POLYMER_GRADE_A  (v2.1)
  Procedure: POLY_BATCH
    Unit Procedure: REACTOR_UP    [allocated to Unit: R-101]
      Operation: CHARGE_OP
        Phase: PURGE_N2_PH        -- inert atmosphere preparation
        Phase: FILL_SOLVENT_PH    -- add solvent to target weight
        Phase: ADD_MONOMER_PH     -- add monomer via mass flow
        Phase: ADD_CATALYST_PH    -- add catalyst via dosing pump
      Operation: REACT_OP
        Phase: HEATUP_PH          -- ramp to 80°C (param: HeatRate_degC_min)
        Phase: HOLD_PH            -- hold at 80°C (param: HoldTime_min)
        Phase: SAMPLE_PH          -- wait for operator to confirm sampling
        Phase: COOL_PH            -- cool to 40°C (param: CoolTemp_degC)
      Operation: DISCHARGE_OP
        Phase: TRANSFER_PH        -- transfer to storage (param: DestTank)
        Phase: HEEL_PH            -- drain residuals
        Phase: RINSE_PH           -- CIP rinse cycle
```

## 3. ISA-88 Phase State Machine

The Phase State Machine is the execution contract. Every Phase, Unit, and Equipment Module must implement this exact state machine. Non-compliant implementations break interoperability with batch management systems.

### States

| State | Description |
|-------|-------------|
| **IDLE** | Phase is inactive and ready to receive START |
| **RUNNING** | Phase is actively executing its programmed action |
| **PAUSING** | Transition: controlled suspension in progress |
| **PAUSED** | Phase is suspended; can RESUME or ABORT |
| **HOLDING** | Transition: moving equipment to safe hold condition |
| **HELD** | Equipment in safe hold state; can RESTART or ABORT |
| **STOPPING** | Transition: executing orderly stop sequence |
| **STOPPED** | Orderly stop complete; ready for RESET |
| **ABORTING** | Transition: executing emergency abort actions |
| **ABORTED** | Emergency abort complete; equipment in safe state |
| **COMPLETE** | Phase finished successfully; awaiting RESET |

### Transitions and Commands

```
IDLE ──[START]──> RUNNING ──[complete]──> COMPLETE ──[RESET]──> IDLE

RUNNING ──[PAUSE]──> PAUSING ──[safe]──> PAUSED ──[RESUME]──> RUNNING
RUNNING ──[HOLD]───> HOLDING ──[safe]──> HELD ──[RESTART]──> RUNNING
RUNNING ──[STOP]───> STOPPING ──[done]──> STOPPED ──[RESET]──> IDLE

PAUSED  ──[STOP]──> STOPPING ...
HELD    ──[STOP]──> STOPPING ...

Any active state ──[ABORT]──> ABORTING ──[done]──> ABORTED ──[RESET]──> IDLE
```

### Hold vs Pause: Critical Distinction

**PAUSE** (operator-initiated controlled suspension):
- Process freezes in its current intermediate state
- Timers stop accumulating
- Equipment stays exactly where it is (valves remain in current position)
- RESUME continues from the exact point where PAUSE was issued
- Use case: operator needs to manually inspect something; batch can safely wait

**HOLD** (safety-driven movement to safe state):
- HOLDING transition code actively drives equipment to a pre-defined safe configuration
- Example: close all feed valves, maintain temperature via jacket (do not let it drift), stop dosing pumps
- HELD state is a specific safe equipment condition, NOT an arbitrary freeze
- RESTART resumes from the RUNNING state logic (may re-execute setup steps within RUNNING)
- Use case: equipment fault (motor trip, valve failure), process limit exceeded, external interlock

**Implementation mistake**: Using PAUSE as an automatic safety response. PAUSE does not drive equipment to a safe state — it merely stops progression. A pressure excursion should trigger HOLD (→ safe configuration), never PAUSE.

### ABORT vs STOP

**STOP**: Orderly conclusion. The phase completes in-progress actions in a controlled sequence before stopping. May take minutes. Leaves equipment ready for a new batch (clean state). Followed by RESET to return to IDLE.

**ABORT**: Emergency termination. Immediately executes abort-safe actions (open relief valves, stop all pumps, initiate emergency discharge). Prioritizes safety over product quality. Followed by RESET only after operator review and confirmation. Aborted batches require disposition decisions (rework, dispose, salvage).

## 4. Recipe Types

ISA-88 defines four recipe types representing different abstraction levels:

| Recipe Type | Scope | Owner | Details |
|-------------|-------|-------|---------|
| **General Recipe** | Any plant, any site | R&D, process engineers | Process-level description: chemistry, quality targets, no equipment specifics |
| **Site Recipe** | Specific site | Site process engineers | Adapted to site capabilities: approved raw material grades, site-specific parameters |
| **Master Recipe** | Specific process cell class | Control engineers | Fully specified: all setpoints, equipment requirements, procedure logic for one process cell |
| **Control Recipe** | Specific batch instance | Batch management system | Created from Master Recipe for one batch: actual equipment assigned, actual Control Recipe ID, timestamp |

### Recipe Content Structure

```
Header:
  RecipeID:        POLY_GRADE_A_v2.1
  ProductID:       P-10042-A
  Version:         2.1
  Status:          Approved
  ApprovedBy:      J.Smith
  ApprovalDate:    2024-01-15
  BatchSize_kg:    500

Formula:
  Solvent_A_kg:    150.0  (±2.0)
  Monomer_B_kg:    120.0  (±0.5)
  Catalyst_C_g:    25.0   (±0.1)

Equipment Requirements:
  ProcessCell:     PC-01
  Unit_Class:      POLYMER_REACTOR
  MinVolume_L:     800
  JacketCapacity:  YES

Procedure: [see section 2 hierarchy]
```

## 5. Control Module and Equipment Module Patterns

### Standard CM Patterns for Process Industry

**On/Off Valve CM:**
```
Inputs (from field):
  OLS (Open Limit Switch), CLS (Closed Limit Switch)

Commands (from EM/sequence):
  OpenCmd, CloseCmd

Mode:
  AUTO  — accept commands from Equipment Module
  MAN   — operator direct command from HMI
  LOCK  — no commands accepted (maintenance)

Status outputs:
  OPEN  (OLS=1, CLS=0)
  CLOSED (CLS=1, OLS=0)
  FAULT (command-feedback mismatch after timeout, typically 15-30s)
  TRANS (transitioning between open and closed)
```

**Motor CM (standard single-speed):**
```
Inputs:
  RunFbk (running feedback), StopFbk (stopped feedback), OL (overload protection)

Commands:
  StartCmd, StopCmd, ResetCmd

Mode:
  AUTO, MAN, LOCK, LOCAL (control from field panel)

Status:
  RUNNING, STOPPED, FAULT, BLOCKED (interlock preventing start)

Alarms:
  StartFail (no RunFbk after start within MonTime, typical 10-30s)
  StopFail  (RunFbk still present after stop command + delay)
  OLTrip    (overload relay actuated)
```

**PID CM:**
```
Inputs:
  PV (process variable, scaled to EU), SP_Ext (external cascade setpoint)

Mode:
  AUTO   (SP from internal recipe setpoint)
  CAS    (SP from external cascade — another controller)
  MAN    (MV directly set by operator)

Parameters:
  Kp, Ki, Kd (or PB%, I_min, D_sec)
  SP_HiLimit, SP_LoLimit (setpoint constraints)
  MV_HiLimit, MV_LoLimit (output constraints)
  AlarmHH, AlarmH, AlarmL, AlarmLL (PV alarms)
```

### Shared EM Arbitration Pattern

```
EM_SolventDosing:
  State: IDLE / BUSY / FAULT
  Owner: which Unit currently has access (NULL if IDLE)

Unit R-101 requests dosing:
  1. R-101 sets Request_R101 := TRUE
  2. Arbitrator checks: if Owner = NULL AND State = IDLE:
       Grant_R101 := TRUE; Owner := R-101; State := BUSY
  3. R-101 executes its dosing phase
  4. R-101 sets Release_R101 := TRUE (phase COMPLETE)
  5. Arbitrator: Owner := NULL; State := IDLE
  6. If Request_R102 was waiting: Grant_R102 := TRUE ...
```

## 6. DCS Mapping: Freelance, PCS7, DeltaV

### ABB Freelance (no native ISA-88 batch layer)

| ISA-88 Concept | Freelance Implementation |
|----------------|------------------------|
| Control Module | MSR point (IDF_1 motor, AIN_1 analog, PID_1 loop) |
| Equipment Module | Logical grouping of MSR points in same node |
| Phase | SFC (Sequential Function Chart) chart in EWP |
| Unit Procedure | Parent SFC calling sub-SFCs |
| Recipe | External batch system via OPC-DA/UA |
| Batch Management | External (AVEVA Batch, custom MES, or none) |

Freelance does not enforce the ISA-88 state machine natively. Engineers must implement RUNNING/HOLDING/PAUSING states manually in SFC logic. The typical approach is a "Phase Shell" SFC with explicit state steps.

### Siemens PCS 7

| ISA-88 Concept | PCS 7 Implementation |
|----------------|---------------------|
| Control Module | CFC chart (one FB per CM, STEP7/CFC editor) |
| Equipment Module | Phase type (SFC with PHASE_AP block) |
| Unit | Plant hierarchy Unit object (S88_UNIT APL block) |
| Phase | SFC with ISA-88 state machine (PHASE_AP provides state management) |
| Recipe | PCS 7 Batch server (separate server application) |
| Motor CM | `MotL` APL block |
| Valve CM | `ValvL` APL block |
| PID CM | `PIDConL` APL block |

PCS 7 enforces ISA-88 state machine compliance through the `PHASE_AP` block which handles all state transitions. Engineers implement the RUNNING logic inside the `RUNNING` step of the SFC.

### Emerson DeltaV

| ISA-88 Concept | DeltaV Implementation |
|----------------|----------------------|
| Control Module | CM (FBD-based, continuous) |
| Equipment Module | EM (state machine + sequential function) |
| Unit | Unit Module (hosts EM hierarchy) |
| Phase | Phase Logic (SFC or Phase steps in DeltaV Batch) |
| Recipe | DeltaV Recipe (Recipe Studio) |
| Motor CM | `DC_MOTOR` or `MOTOR_CTRL` Device Control block |
| Valve CM | `DC_VALVE` Device Control block |
| PID CM | `PID` or `PID_LOOP` block |

DeltaV's Device Control (DC) block implements modes (AUTO/MAN/LOCKED) and the RUNNING/STOPPED/FAULT state machine. Phase Input Attributes (PIAs) pass recipe setpoints to phases; Phase Output Attributes (POAs) return results.

## 7. ISA-95 Enterprise-Control Integration

ISA-95 Part 1 defines the conceptual model. Part 2 defines the object model. Part 3 defines activity models. Parts 4-5 define data structures for B2MML.

### ISA-95 Information Categories

**Personnel Information**: Who is qualified/authorized to operate equipment, perform procedures.

**Equipment Information**: Equipment capabilities, current status, availability. Maps to ISA-88 physical model.

**Material Information**: Raw material specifications, lot information, inventory levels.

**Process Segment Information**: Sequence of operations needed for production. Maps to ISA-88 procedural model.

**Production Capability**: What can be produced, at what rate, with what constraints.

**Production Schedule**: ERP → MES: what to produce, when, how much, with which materials.

**Production Performance**: MES → ERP: what was actually produced, when, with what quality results.

**Quality Test Information**: LIMS results associated with production batches.

### B2MML (Business To Manufacturing Markup Language)

XML implementation of ISA-95 data structures. Standard messages:

```xml
<!-- Production Order from ERP to MES -->
<ProductionOrder xmlns="http://www.mesa.org/xml/B2MML">
  <ID>PO-2024-001234</ID>
  <Description>Polymer Grade A - 500 kg</Description>
  <ProductionVersion>POLY_GRADE_A_v2.1</ProductionVersion>
  <StartTime>2024-03-14T06:00:00</StartTime>
  <EndTime>2024-03-14T14:00:00</EndTime>
  <PublishedDate>2024-03-13T16:00:00</PublishedDate>
  <ProductionOrderSegment>
    <ID>SEG-001</ID>
    <ProcessSegmentID>POLY_BATCH_SEG</ProcessSegmentID>
    <MaterialProducedActualQuantity unitOfMeasure="KG">
      <QuantityString>500</QuantityString>
    </MaterialProducedActualQuantity>
  </ProductionOrderSegment>
</ProductionOrder>
```

## 8. Purdue Reference Model (PERA)

The Purdue Reference Model defines five levels of industrial control hierarchy, widely adopted as the basis for industrial cybersecurity zoning:

```
Level 4: Enterprise (ERP/Business Systems)
    SAP, Oracle, business planning, supply chain, financial systems
    Typical systems: SAP PM, SAP PP, Oracle EBS

Level 3: Site Operations (MES/LIMS/Batch)
    Manufacturing execution, production scheduling, quality systems
    Typical systems: DeltaV Batch, AVEVA Batch, SAP MES, LIMS, Historian

Level 2: Supervisory Control (SCADA/DCS Operator)
    Process overview, trending, alarm management, operator commands
    Typical systems: DCS Operator Stations, SCADA, Historian interfaces

Level 1: Basic Control (DCS/PLC)
    Regulatory control loops, sequence control, motor/valve management
    Typical systems: DCS Controllers (AC700F, AS 410, DeltaV M-series), PLCs

Level 0: Field Devices
    Physical measurement and actuation
    Typical devices: Transmitters (pressure, flow, temperature, level), Control valves, Motor starters
```

### Security Zones (IEC 62443 / ISA-99)

Each Purdue level is a distinct security zone:
- **Zone 4** (Level 4): Corporate network — least restrictive
- **Zone 3** (Level 3): Operations network — restricted access
- **DMZ**: No direct L3-L4 communication; data replication via historian or message broker
- **Zone 2** (Level 2): Control network — no internet access
- **Zone 1** (Level 1): Controller network — no workstation-class systems
- **Zone 0** (Level 0): Field device network — dedicated fieldbus (PROFIBUS, HART, Foundation Fieldbus)

Direct Level 0↔Level 4 or Level 1↔Level 4 communication is prohibited in secure IEC 62443-compliant architectures.

## 9. ISA-88 + ISA-95 Integration Flow

```
[ERP - Level 4]
  1. Production planner creates Production Order in SAP/Oracle
  2. Production Order sent to MES via B2MML XML or SAP IDOC
  ↓

[MES - Level 3]
  3. MES receives Production Order
  4. Selects Master Recipe version (POLY_GRADE_A_v2.1)
  5. Checks equipment availability (Unit R-101 IDLE?)
  6. Creates Control Recipe instance with:
     - Unique Batch ID: BATCH-2024-001234
     - Equipment Assignment: Process Cell PC-01, Unit R-101
     - Actual parameters from formula
  7. Sends Work Request to DCS Batch Management
  ↓

[DCS Batch / Level 2-3]
  8. Receives Control Recipe
  9. Allocates Unit R-101 (sets Unit state to ALLOCATED)
  10. Executes Procedure → Unit Procedure → Operations → Phases
  11. Each Phase executes via ISA-88 state machine
  12. Records actual values: temperatures reached, quantities added, timestamps
  ↓

[DCS Batch → MES]
  13. Reports Phase completion events with actuals
  14. Reports Batch COMPLETE or ABORT
  15. Sends material consumption actuals
  ↓

[MES → ERP]
  16. MES sends Production Performance to ERP (B2MML)
  17. ERP closes Production Order
  18. Updates inventory: raw materials consumed, product produced
  19. Generates warehouse transfer order
```

## 10. Corner Cases and Implementation Details

### Phase Parameter Timing (DeltaV)

In DeltaV, Phase Input Attributes (PIAs) are sent to the phase **before** the START command. The phase must acknowledge all PIAs before executing. If START arrives before all PIAs are acknowledged, DeltaV queues the start and waits for PIA acknowledgment. Never rely on parameters set in a previous batch run — always explicitly set all PIAs for each new batch.

### Unit Allocation Conflicts

The ISA-88 standard requires a Unit to refuse allocation if it is not IDLE. In practice, DCS implementations must provide:
1. Unit availability check before allocating
2. Exclusive lock mechanism (only one recipe can allocate a Unit at a time)
3. Clear "Unit Occupied" status visible to operators
4. Rollback mechanism if allocation fails partway through (e.g., Unit A allocated OK, Unit B fails — must release Unit A)

### COMPLETE vs STOP: Clean vs Interrupted Batches

A batch that reaches COMPLETE in all phases is a "clean batch" — traceable, quality test possible. A batch that reaches STOPPED (operator-commanded stop mid-batch) is an "interrupted batch" — may be reworked. A batch that reaches ABORTED is an "emergency-terminated batch" — requires full material disposition (typically disposed). MES and ERP handle these three outcomes differently.

### Recipe Version and Batch Genealogy

Control Recipes must record the exact Master Recipe version used. If the Master Recipe is updated after a Control Recipe was created, the Control Recipe continues using its original version. This is critical for:
- Regulatory compliance (pharmaceutical, food) requiring exact reproduction of approved recipe
- Root cause analysis when product fails quality tests
- Audit trails required by FDA 21 CFR Part 11 or EU GMP Annex 11

### Shared EM Deadlock

If Unit A holds EM_X and is waiting for EM_Y, while Unit B holds EM_Y and is waiting for EM_X, a deadlock occurs — both batches halt indefinitely. The ISA-88 standard does not specify a deadlock resolution mechanism. Practical solutions:
- Design equipment topology to avoid circular dependencies
- Implement a timeout on EM request (HOLD after N minutes of waiting)
- Use a centralized resource arbiter that detects and breaks deadlocks
- Restrict batch scheduling to prevent simultaneous allocation requests for the same EM pair

### ISA-95 Latency Tolerance

ERP systems may have hours of downtime for maintenance. The MES and DCS must be capable of operating for at least 24-72 hours without ERP connectivity:
- Buffer production orders received before downtime
- Continue executing batches
- Queue batch reports locally
- Reconcile with ERP upon reconnection

The control system must NEVER wait for an ERP response before executing a batch step. The ISA-95 interface is asynchronous by design.

### ISA-88 Applied to Continuous Processes

ISA-88 was designed for discrete batch operations. Applying it to continuous processes (no discrete batch boundary) requires defining "virtual batches" — production periods treated as batches:
- Time-defined: every 8-hour shift is one "batch"
- Volume-defined: every N tonnes produced is one "batch"
- Recipe-change-defined: a batch starts when product grade changes and ends at next change

This is common in continuous blending, paper machines, and continuous polymerization where the concept of "batch" is contractual rather than physical.
