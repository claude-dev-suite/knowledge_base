# ISA-18.2 / ISA-101 - Alarm Management and HMI Design

> Official Reference: https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa18
> ISA-101 Reference: https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa101
> Industry Reference: https://www.emerson.com/documents/automation/white-paper-alarm-management-for-process-control-en-68764.pdf

## Overview

ISA-18.2 (ANSI/ISA-18.2-2016) defines the management lifecycle of alarms in process control systems. It establishes the definition of an alarm, alarm priority classification, performance KPIs, rationalization methodology, and ongoing management processes.

ISA-101 (ANSI/ISA-101.01-2015) defines the philosophy, design, implementation, operation, and management of HMI (Human-Machine Interface) systems for industrial process control. Together, ISA-18.2 and ISA-101 define what an operator should see and how the system should alert them.

## 1. ISA-18.2 Alarm Definitions and Classification

### What Is an Alarm?

ISA-18.2 defines an alarm as:
> "An audible and/or visible means of indicating to the operator an equipment malfunction, process deviation, or abnormal condition requiring a response."

**Three required elements of a true alarm:**
1. **Indication**: the operator is notified (visual and/or audible)
2. **Condition**: there is an equipment malfunction, process deviation, or abnormal condition
3. **Response required**: the operator must take a specific action

An indicator without a required response is a **message** (or event), not an alarm. A process variable exceeding a threshold that the operator cannot act on is not a proper alarm — it should be an event/message or the limit should be moved.

### Alarm vs Event

| Characteristic | Alarm | Event |
|----------------|-------|-------|
| Operator response | Required | Not required |
| Acknowledgment needed | Yes | No (logged only) |
| Priority | Yes (1-4) | No |
| Silenceable | After acknowledgment | N/A |
| Examples | High temperature (requires operator action) | Mode change confirmation, setpoint change record |

### Alarm States

An alarm can be in one of these states at any time:

| State | Description |
|-------|-------------|
| **Normal** | Alarm condition not present; no indication |
| **Unacknowledged Active** | Condition is present; operator has NOT acknowledged (most critical state — needs immediate attention) |
| **Acknowledged Active** | Condition is present; operator HAS acknowledged (knows about it) |
| **Unacknowledged Return to Normal (RTN)** | Condition cleared while still unacknowledged; operator must acknowledge the occurrence |
| **Suppressed** | Alarm has been temporarily removed from service (see section 7) |
| **Shelved** | Alarm temporarily removed by operator for a defined time period |
| **Out of Service (OOS)** | Alarm disabled for maintenance |
| **Designed Suppressed** | Alarm disabled by engineering logic (process-state-based suppression) |

## 2. Alarm Priority Framework

ISA-18.2 defines four standard alarm priorities. The priority determines:
- The maximum time an operator has to respond before a consequence occurs
- The visual/audible presentation (color, flash rate, sound)
- The percentage of total alarms at that priority (target distribution)

### Priority Definitions

| Priority | Level | Response Time | Consequence if No Response | Target % of Alarms |
|----------|-------|---------------|---------------------------|-------------------|
| **1 - Critical** | Highest | Immediate (< 5 minutes) | Catastrophic: personnel injury, major environmental release, major equipment damage, emergency shutdown | < 5% |
| **2 - High** | | < 15 minutes | Significant: reduced product quality, minor equipment damage, production loss > 30 minutes | < 15% |
| **3 - Medium** | | < 30 minutes | Moderate: degraded operation, production rate reduction, quality deviation | < 30% |
| **4 - Low** | Lowest | < 60 minutes | Minor: informational; operator should be aware but immediate action not critical | ~50% |

**Target distribution**: approximately 5% Critical, 15% High, 30% Medium, 50% Low.

Plants with more than 20-25% of alarms in Priority 1-2 are alarm-flooded systems — operators cannot respond effectively.

### Typical Priority Assignments

| Condition | Typical Priority |
|-----------|-----------------|
| Safety Instrumented System pre-alarm | 1 (Critical) |
| Process high-high / low-low (close to SIS trip setpoint) | 1 (Critical) |
| Equipment failure (motor trip, valve failure) affecting production | 2 (High) |
| Process high / low (first deviation from target range) | 3 (Medium) |
| Soft sensor mismatch, maintenance required | 4 (Low) |
| Communication failure to non-critical device | 4 (Low) |
| Manual mode active | 4 (Low) or message |

## 3. Alarm Lifecycle

ISA-18.2 defines a full lifecycle for alarm management — from initial design through decommissioning.

### Lifecycle Stages

```
1. Identification
   → Identify process hazards, equipment failures, abnormal conditions
   → Source: P&ID review, HAZOP, FMEA, process knowledge
   → Output: candidate alarm list

2. Rationalization
   → Evaluate each candidate alarm: is it necessary? right priority? right setpoint?
   → Apply LOPA (Layer of Protection Analysis) to priority determination
   → Document: cause, consequence, operator response, priority, setpoint, deadband
   → Output: rationalized alarm database

3. Detailed Design
   → Configure alarm in DCS: setpoint, deadband, on-delay, off-delay, priority
   → Define suppression logic (state-based suppression)
   → Design HMI presentation
   → Output: DCS alarm configuration

4. Implementation
   → Load alarm configuration into DCS
   → Test each alarm (loop check, alarm verification)
   → FAT (Factory Acceptance Test) includes alarm verification

5. Operation
   → Monitor alarm KPIs continuously
   → Identify bad actors (chattering, fleeting, stale)
   → Respond to alarms per response guidelines

6. Maintenance
   → Review and update alarm setpoints as process conditions change
   → Re-rationalize alarms that are repeatedly acknowledged without action
   → Modify or remove nuisance alarms

7. Monitoring and Assessment
   → Regular (monthly) KPI review against ISA-18.2 benchmarks
   → Annual alarm performance audit
   → Continuous improvement program

8. Decommissioning
   → Remove alarms when the associated equipment is decommissioned
   → Archive alarm history for regulatory compliance
```

## 4. Alarm KPIs and Performance Targets

ISA-18.2 defines measurable performance targets for alarm systems. These are monitored through the Alarm Management System (AMS) or DCS historian.

### Key Performance Indicators

| KPI | Definition | ISA-18.2 Target | "Overloaded" Threshold |
|-----|-----------|-----------------|----------------------|
| **Average alarm rate** | Alarms per operator per 10 minutes (steady state) | < 1 alarm / 10 min per operator | > 10 alarms / 10 min |
| **Peak alarm rate** | Maximum alarms in any 10-minute window | < 10 alarms / 10 min | > 100 alarms / 10 min |
| **Average alarms standing** | Alarms active at any one time | < 5-10 per operator | > 30 |
| **Unacknowledged alarm average** | Unacknowledged alarms outstanding | < 5 per operator | > 25 |
| **Chattering alarm count** | Alarms activating > 3 times in 24 hours | 0% of alarm list | > 1% |
| **Stale (fleeting) alarm count** | Alarms active > 24 hours continuously | 0% of alarm list | > 5% |
| **Shelved alarms** | Alarms currently shelved | < 1% of total | > 5% |
| **Priority 1 response time** | Time to acknowledge Priority 1 alarms | 100% < 5 minutes | — |

### Annunciator Loading Classification (EEMUA 191)

The EEMUA 191 guide (complementary to ISA-18.2) classifies alarm loading:

| Alarm Rate | Classification |
|-----------|----------------|
| < 1/10 min | Acceptable |
| 1-2/10 min | Manageable |
| 2-6/10 min | Very Busy |
| > 6/10 min | Overloaded — operator cannot respond effectively |

### Monitoring Tools

- **Alarm Frequency Analysis**: rank all alarms by frequency over rolling 30/90/365 days
- **Bad Actor Report**: top 10 most frequent alarms (usually responsible for 60-80% of all alarm activations)
- **Flood Analysis**: identify time periods with alarm rates > threshold
- **Acknowledgment Time Tracking**: ensure Priority 1 alarms are acknowledged within 5 minutes

## 5. Alarm Rationalization

Rationalization is the process of evaluating each alarm to confirm it is necessary, properly configured, and actionable.

### Rationalization Record (per alarm)

Each alarm must have a documented rationalization record:

| Field | Description |
|-------|-------------|
| Tag ID | Instrument tag (e.g., TIC-101A_HH) |
| Alarm type | High/Low/High-High/Low-Low/Deviation/Rate-of-Change |
| Cause | What process condition triggers this alarm |
| Consequence | What happens if the operator does not respond |
| Response | Specific action operator must take (not generic "investigate") |
| Time available | Maximum time before consequence if no response |
| Priority | 1-4, justified by response time |
| Setpoint | Trip value with units |
| Deadband | Width to prevent chattering on limit crossing |
| On-delay | Minimum time condition must persist before alarm activates |
| Off-delay | Minimum time condition must clear before alarm deactivates |
| Suppression conditions | Process states where alarm should be suppressed |

### LOPA Integration

Layer of Protection Analysis (LOPA) can justify alarm priority:
- If the alarm is the only layer of protection before a hazardous event → Priority 1
- If the alarm is an early warning with 3 other independent layers → Priority 3-4
- Alarms identified as safeguards in a HAZOP should be Priority 1 or 2

### Rationalization Process

1. Form cross-functional team: Process Engineer, Control Engineer, Operations, Safety
2. Review each alarm against rationalization criteria (is it necessary? actionable? right priority?)
3. Remove alarms with no operator response (convert to events/messages)
4. Combine duplicate alarms for the same condition
5. Set setpoints using engineering justification, not "engineer's best guess"
6. Document decisions with traceability to HAZOP/LOPA references

## 6. Bad Actors: Chattering, Fleeting, Stale Alarms

### Chattering Alarms

**Definition**: An alarm that activates and deactivates rapidly, multiple times within a short period (typically > 3 times in 10 minutes), due to the process value oscillating across the alarm setpoint.

**Root causes**:
- Setpoint too close to normal operating value
- Deadband too small relative to process noise
- Noisy sensor signal
- Process instability / oscillating control loop
- Wrong alarm type for the condition (e.g., using absolute limit for a cyclic process)

**Solutions**:
- Increase deadband (separate activation threshold from deactivation threshold)
- Move setpoint further from normal operating point (if safety allows)
- Add on-delay (require condition to persist for N seconds before activating)
- Fix underlying process instability (tune PID, resolve sensor noise)
- Use rate-of-change alarm instead of absolute limit for trends

### Fleeting Alarms

**Definition**: Alarms that activate briefly (< 30 seconds) and then return to normal, often before the operator can respond.

**Causes**: Process transients, momentary sensor glitches, brief equipment state changes.

**Solutions**:
- Add on-delay: alarm activates only if condition persists for N seconds
- Review if the alarm is actually necessary (if it self-corrects, is operator action needed?)
- Convert to event/message if no operator action is required

### Stale Alarms

**Definition**: Alarms that have been active continuously for an extended period (> 24 hours or > 72 hours depending on organization's definition). Operators begin to ignore them ("alarm fatigue").

**Causes**:
- Alarm setpoint is wrong (process has permanently moved; setpoint was never updated)
- Equipment is in a non-standard operating mode that was not accounted for
- Sensor failed and alarm was not properly managed (should be OOS)
- Alarm was configured for a condition that is now normal (process change, product change)

**Solutions**:
- Audit all alarms active > 24 hours daily
- Re-rationalize: is the setpoint still correct? Is the condition actually abnormal?
- If condition is expected in current operating mode: add state-based suppression
- If sensor failed: declare OOS, arrange maintenance, suppress alarm during OOS

### Nuisance Alarms

A broader category including chattering, fleeting, and stale alarms, plus:
- **Consequential alarms**: multiple alarms triggered by one root cause (should be suppressed once root cause alarm fires)
- **Out-of-service alarms**: alarms firing because equipment is legitimately down for maintenance (should be suppressed during OOS periods)
- **Mode-dependent alarms**: alarms that are inappropriate in certain operating modes

## 7. Alarm Suppression, Shelving, and Inhibiting

ISA-18.2 distinguishes three distinct methods for removing an alarm from service. They differ in authorization level, duration, and record-keeping requirements.

### Suppression (Designed Suppression)

**Definition**: Automatic, programmatic removal of an alarm by engineering logic based on process state.

**Characteristics**:
- Designed and configured during engineering (rationalization)
- No operator action required
- Activates automatically when suppression condition is met
- Documented in the rationalization record
- Examples:
  - Suppress "pump not running" alarm when pump is in MANUAL mode
  - Suppress "low flow" alarm when feed valve is closed
  - Suppress "high pressure" alarm during startup sequence (transient condition expected)

### Shelving

**Definition**: Temporary removal by the operator for a specific, limited time period.

**Characteristics**:
- Operator action (conscious decision)
- Time-limited: shelving expires automatically after the configured maximum duration (typically 8-24 hours)
- Requires operator acknowledgment and reason code
- Automatically re-activates when time expires or when operator un-shelves
- All shelving actions are logged with timestamp, operator ID, duration, and reason
- Should be used sparingly (ISA-18.2 target: < 1% of alarms shelved at any time)

**Appropriate use**: A sensor is known to be temporarily unreliable (e.g., maintenance is being done on the instrument, but the process must continue). The alarm is shelved until maintenance is complete.

**Inappropriate use**: Shelving a chattering alarm indefinitely instead of fixing the underlying engineering issue.

### Inhibiting (Out of Service)

**Definition**: Removal of an alarm during maintenance, equipment outage, or other planned periods.

**Characteristics**:
- Requires supervisor or engineer authorization (higher than shelving)
- Not time-limited (persists until explicitly re-enabled)
- Must be fully documented in change management system
- All inhibit/restore actions logged with authorization trail

**Appropriate use**: A transmitter has failed and will be replaced tomorrow. The instrument is declared "Out of Service" and all alarms associated with it are inhibited until the replacement is commissioned.

## 8. DCS Alarm Implementation (ABB Freelance)

In ABB Freelance, alarms are configured at the MSR (Measurement/Signal Record) level through the EAM (External Access Manager) and the function block parameters.

### Alarm Naming Convention (Freelance)

| Signal Suffix | Alarm Type | Description |
|---------------|------------|-------------|
| `XA1{node}` | Primary fault alarm | Motor trip, valve failure, protection activation |
| `XA2{node}` | Secondary alarm / SDS | Shutdown or interlock alarm |
| `XAH{tag}` | High alarm | Analog high alarm |
| `XAHH{tag}` | High-high alarm | Analog high-high alarm |
| `XAL{tag}` | Low alarm | Analog low alarm |
| `XALL{tag}` | Low-low alarm | Analog low-low alarm |

### Alarm Properties in Freelance

Each alarm in Freelance can be configured with:

| Property | Description | Typical Value |
|----------|-------------|---------------|
| Priority | 1-4 per ISA-18.2 | Derived from rationalization |
| SP (Setpoint) | Alarm trip value (for analog alarms) | Process-specific |
| Deadband | Hysteresis to prevent chattering | 1-5% of range |
| On-delay | Time condition must persist before activation | 0-60 seconds |
| Off-delay | Time condition must clear before deactivation | 0-30 seconds |
| Class | Alarm class (A=process, B=equipment, C=communication) | A, B, or C |
| Shelving | Enable/disable shelving by operator | Typically enabled for low priority |
| Suppress | Link to suppression condition (Boolean MSR signal) | State-based logic |

### Alarm EAM Record Example

```
(* Motor XA1 alarm — primary fault *)
[EAM:RECORD];1;XA111301CLWW1A1;0;BOOL;MOTOR FAULT ALARM;1;1
  Priority    := 2       (* HIGH - motor trip stops production *)
  On_Delay    := 0       (* Immediate *)
  Off_Delay   := 5       (* 5 second hold to confirm clearance *)
  Deadband    := 0       (* Binary signal: no deadband needed *)
  Class       := B       (* Equipment alarm *)

(* Analog HH alarm *)
[EAM:RECORD];1;XAHH11301FIC056A;0;BOOL;FLOW HIGH-HIGH;1;1
  Priority    := 1       (* CRITICAL - safety instrumented function pre-alarm *)
  SP          := 1050.0  (* m3/h *)
  Deadband    := 20.0    (* m3/h - prevents chattering near 1050 *)
  On_Delay    := 5       (* seconds - confirm condition is real *)
  Off_Delay   := 10      (* seconds *)
  Class       := A       (* Process alarm *)
```

### Alarm Suppression Logic in Freelance

State-based suppression is implemented by connecting a Boolean signal to the alarm's suppression input:

```
Suppress_XA1_Motor := (Mode_MANUAL OR Mode_LOCAL OR State_STOPPED);

(* When motor is in manual or local mode, or confirmed stopped,
   the running feedback fault alarm is suppressed.
   Otherwise an operator stopping the motor manually would
   generate a nuisance fault alarm. *)
```

## 9. ISA-101 HMI Design Standard

ISA-101 defines the philosophy and design standards for process HMI systems. Its core principle is **situation awareness**: the operator must be able to understand the current process state, detect deviations, and determine required responses quickly.

### ISA-101 Core Design Philosophy

**The High-Performance HMI Philosophy** (formalized in ISA-101, originally from Abnormal Situation Management Consortium):

1. **Gray-dominant background**: Use neutral gray (not black, not white) as the primary background color. Colored backgrounds are visually fatiguing and reduce contrast for abnormal indications.

2. **Muted, desaturated normal state colors**: In normal operation, displays should use desaturated tones (gray-blue, gray-green). Vivid colors are reserved exclusively for abnormal conditions.

3. **Color conveys state, not identity**: A red indicator means "abnormal" — NOT "this is a pump" or "this is a temperature". Using red for all temperature elements destroys the alarm signaling value of red.

4. **No decoration for decoration's sake**: 3D effects, gradients, drop shadows, and decorative graphics add no information and reduce scan speed. Use flat, 2D representations.

5. **Operator-centered layout**: Arrange displays based on operator workflow and decision-making process, not on P&ID layout or equipment physical layout.

6. **Consistent navigation**: Navigation between display levels must be consistent and predictable. Operators should not need to search for how to reach a detail display.

7. **Meaningful symbology**: Equipment symbols should convey state information (running/stopped/faulted), not just identity.

## 10. HMI Display Hierarchy

ISA-101 defines four display levels (plus an additional level for some implementations):

### Level 1: Plant Overview

**Purpose**: High-level view of the entire plant or major process area. Allows operators to see the overall health of the plant at a glance.

**Content**:
- Process overview with key status indicators (running/stopped/alarm states)
- Overall production KPIs (throughput, yield, energy)
- Critical alarm summary
- No individual instrument readings (too much detail)
- Major equipment state indicators (color-coded: normal/abnormal)

**Operator use**: First display seen at shift handover. Identifies which areas need attention without navigating into details.

### Level 2: Process Unit / Area Display

**Purpose**: Main working display for the operator managing a specific process unit or area. Shows all process parameters needed for active monitoring and control.

**Content**:
- P&ID-style layout of the unit (simplified, not the full P&ID)
- All key process variables with current values and trend indicators
- Equipment states (running/stopped/fault with color coding)
- Alarm indicators (only abnormal alarms highlighted)
- Quick access to setpoints for PID loops
- Key interlocks and permissives

**Operator use**: The primary display for continuous monitoring. Operator spends most time here.

### Level 3: Unit/Group Display

**Purpose**: Detailed view of a specific equipment group or subprocess. Called when investigating an anomaly identified at Level 2.

**Content**:
- Detailed process flow for one specific unit (e.g., all feeds to one reactor)
- All instrument readings (including those not shown at Level 2)
- Historical trends for key variables (last 1-4 hours)
- Detailed alarm list for this unit
- PID controller faceplates

**Operator use**: Investigation display. Not continuously monitored.

### Level 4: Faceplates and Diagnostics

**Purpose**: Individual instrument or control loop detail. Used for configuration and diagnostics.

**Content**:
- PID faceplate: PV, SP, MV, mode (AUTO/MAN/CAS), tuning parameters
- Motor faceplate: running/stopped/fault/mode status, start/stop commands, runtime hours
- Valve faceplate: open/closed/position, commands, fault indicators
- Transmitter faceplate: raw value, scaled value, HART diagnostics
- Alarm configuration (setpoints, priorities, shelving status)

**Operator use**: On-demand when manipulating a specific control element or investigating a specific device.

## 11. Color, Navigation, and Display Design Rules

### Color Standards (ISA-101)

**Background colors:**
- Primary background: **Medium gray** (RGB ~160,160,160 or similar neutral tone)
- Secondary background (panels, faceplates): Slightly lighter gray

**Normal state process elements:**
- Pipes: Gray-blue (flowing fluid) or gray (empty/idle)
- Equipment outlines: Dark gray
- Normal running motors: Desaturated green or dark gray — NOT bright green
- Normal closed valves: Dark filled symbol — NOT red
- Text labels: Dark gray or black

**Abnormal state (use bright colors ONLY for abnormal conditions):**
- **Red**: High priority alarm active (Priority 1-2), equipment in fault state
- **Yellow/Amber**: Medium priority alarm, warning condition (Priority 3)
- **Magenta/Pink**: Low priority alarm, advisory (Priority 4) — some implementations
- **Flashing**: Unacknowledged alarm (flash stops after acknowledgment)
- **Cyan/White**: Shelved or suppressed alarm (not actively alarming)

**Mode indicators (not alarms — use desaturated tones):**
- Manual mode: Yellow-gold (desaturated, NOT the same yellow as alarms)
- Cascade mode: Cyan-teal
- Out of service: Gray with X or strikethrough

### Navigation Rules

1. **Consistent navigation path**: Every display must have a clearly labeled "back" or "up" button returning to the Level 2 display.
2. **Display title**: Every display must show its name, the area it covers, and the current date/time.
3. **Maximum 2 clicks** to reach any faceplate from the Level 2 area display.
4. **No orphan displays**: Every display must be reachable from the standard navigation hierarchy (no display only accessible from one specific other display).
5. **Alarm banner**: The global alarm banner (showing total alarm count, highest priority unacknowledged) must be visible from every display.

### Analog Value Presentation

Analog process values should be displayed with:
- A numerical value with engineering units
- A context indicator (operating range, normal band, alarm setpoints)
- A trend sparkline or bar graph showing recent direction
- Alarm state coloring on the value (red/yellow/normal) per the active alarm state

**Bad practice**: Showing a process value in a large red number as a design decision (the number is always red for "temperature"). Red should ONLY appear when an alarm is active.

### Dynamic Process Graphics

For equipment, use dynamic state symbols:
- **Motor running**: Rotating symbol, green indicator — NOT just green text
- **Motor stopped**: Stationary symbol, gray or dark indicator
- **Motor faulted**: Red indicator, fault symbol
- **Valve open**: Symbol shows open gate/disc
- **Valve closed**: Symbol shows closed gate/disc, filled symbol
- **Valve transitioning**: Animated or amber indicator

Dynamic symbology dramatically reduces the cognitive load on operators compared to text-only status indicators.

## 12. Corner Cases and Non-Obvious Requirements

### Acknowledging vs Silencing

Acknowledging an alarm and silencing the audible alert are different operations. ISA-18.2 requires:
- **Acknowledgment**: operator confirms they have seen and are responding to the alarm
- **Silence**: mutes the audible alert only (the visual indication remains)

A system that silences ALL alarms with one button (common in older systems) is non-compliant because it removes the audible distinction between the silenced alarm and any new alarms.

### Unacknowledged Return to Normal (RTN)

When an alarm condition clears before the operator acknowledges it, the alarm enters the "Unacknowledged RTN" state. This state must remain visible to the operator — the process condition went abnormal and returned without the operator's knowledge. The operator must acknowledge the RTN to clear it. Automatically clearing unacknowledged RTN alarms hides transient process events.

### Priority 1 Alarm Staffing

ISA-18.2 implies that every Priority 1 alarm must be responded to within the defined response time. If the response time is 5 minutes for Priority 1, the operator staffing must be sufficient to respond to any combination of Priority 1 alarms within 5 minutes. An alarm system with 200 Priority 1 alarms that could all activate simultaneously is not manageable regardless of the response time specification.

### Setpoint and Deadband Interaction

Deadband is the width of the hysteresis band around the setpoint. If SP = 100 and Deadband = 5:
- Alarm activates when value **rises above 105** (setpoint + deadband)
- Alarm deactivates when value **falls below 95** (setpoint - deadband)

Wait — this varies by implementation. In many DCS systems:
- Alarm SP is the activation threshold
- Alarm deactivates when value returns to SP minus Deadband

In ABB Freelance specifically:
- For high alarm: activates at SP, deactivates at SP - Deadband
- For low alarm: activates at SP, deactivates at SP + Deadband

Always verify the specific DCS behavior, as "deadband" implementations vary between vendors.

### Alarm Flood Duration

During a process upset (emergency, startup, shutdown), alarm rates may exceed the ISA-18.2 targets by 10x or more. This is expected and acceptable for limited durations. ISA-18.2 defines this as an "alarm flood" and recognizes it as a special operating condition. The requirement is that the alarm system help the operator identify the root cause alarm (the first alarm that triggered the cascade) — this is the "first-out" or "cause alarm" concept. Systems that present 200 simultaneous alarms without identifying the root cause are not useful during flood conditions.

### First-Out Alarm Logic

In a well-designed system, when Equipment A fails and causes Equipment B, C, and D to also fault:
- The Equipment A alarm should be Priority 1 (root cause)
- Alarms for B, C, D should be suppressed while A is in fault (they are consequences, not independent events)
- When A is repaired and reset, the B, C, D suppression is removed

This is "first-out" logic, sometimes called "cause and effect" alarm management. Implementation requires state-based suppression in the DCS.

### ISA-101 and Dark-Mode vs Light-Mode Debate

The ISA-101 gray background is the **medium gray** standard, not a black "dark mode". High-performance HMIs on gray backgrounds outperform both black backgrounds (too much glare from white text, poor contrast in bright control rooms) and white backgrounds (fatigue in dimly lit control rooms, red on white is hard to distinguish). The gray background was validated empirically by the Abnormal Situation Management Consortium before becoming an ISA standard.

### Alarm Rate During Major Plant Transitions

During plant startup and shutdown, alarm rates naturally spike as the process moves through abnormal conditions that are expected and not actionable. ISA-18.2 requires implementing "state-based alarming" — different alarm setpoints or suppressions for different plant operating states:
- STARTUP: many alarms suppressed or have modified setpoints
- NORMAL OPERATION: standard alarm configuration
- SHUTDOWN: different suppressions for shutdown state
- MAINTENANCE: OOS suppression for instruments under maintenance

Not implementing state-based alarming causes thousands of nuisance alarms during startup/shutdown, training operators to ignore alarms.
