# Windows Drivers - KMDF - Overview

> **Source**: https://learn.microsoft.com/en-us/windows-hardware/drivers/wdf/using-the-framework-based-on-the-driver-type
> **Skill**: dev-suite skill `windows/wdf-kmdf` — see SKILL.md for the always-loaded quick reference.

## What this covers

Why Microsoft built the Windows Driver Frameworks (KMDF/UMDF) on top of WDM, the KMDF object model and parent/child lifetime rules, and the high-level state machine that wraps the IRP plumbing you would otherwise hand-roll. Read this when you need to make architectural decisions about driver type (function vs filter, FDO vs PDO) or when something the framework "magically" handles is misbehaving.

## Deep dive

### Why WDF over raw WDM

A WDM driver is a `DRIVER_OBJECT` plus a hand-written dispatch table (`MajorFunction[IRP_MJ_*]`), and you implement every PnP minor yourself: `IRP_MN_START_DEVICE`, `IRP_MN_QUERY_REMOVE_DEVICE`, `IRP_MN_SURPRISE_REMOVAL`, etc. The result is hundreds of lines of `IoSkipCurrentIrpStackLocation` / `IoCallDriver` / completion-routine plumbing per driver.

KMDF replaces that with a state machine inside `Wdf01000.sys`. You register typed callbacks (`EvtDevicePrepareHardware`, `EvtDeviceD0Entry`, `EvtIoDeviceControl`); the framework owns the IRPs, enforces ordering, drains queues on remove, and reference-counts your objects. You write only the device-specific behaviour.

### The object hierarchy

Every WDF handle (`WDFDRIVER`, `WDFDEVICE`, `WDFQUEUE`, `WDFREQUEST`, `WDFTIMER`, `WDFWORKITEM`, `WDFMEMORY`...) is opaque and reference-counted. Objects form a tree:

```
WDFDRIVER
 └── WDFDEVICE (one per AddDevice)
      ├── WDFQUEUE (default + extras)
      │    └── WDFREQUEST (transient)
      ├── WDFINTERRUPT
      ├── WDFTIMER
      └── WDFFILEOBJECT (per CreateFile)
```

Destroying a parent destroys all descendants. You can override the parent at create time via `WDF_OBJECT_ATTRIBUTES.ParentObject` — useful for binding a `WDFTIMER` to a request so it dies with the request.

### Per-object context blobs

Instead of `DEVICE_OBJECT->DeviceExtension`, you attach a typed struct via `WDF_OBJECT_ATTRIBUTES_SET_CONTEXT_TYPE`:

```c
typedef struct _MY_DEVICE_CONTEXT {
    WDFQUEUE  ManualQueue;
    ULONG     PendingCount;
    WDFSPINLOCK Lock;
} MY_DEVICE_CONTEXT, *PMY_DEVICE_CONTEXT;
WDF_DECLARE_CONTEXT_TYPE_WITH_NAME(MY_DEVICE_CONTEXT, GetDeviceContext)

WDF_OBJECT_ATTRIBUTES attrs;
WDF_OBJECT_ATTRIBUTES_INIT_CONTEXT_TYPE(&attrs, MY_DEVICE_CONTEXT);
WdfDeviceCreate(&DeviceInit, &attrs, &device);
PMY_DEVICE_CONTEXT ctx = GetDeviceContext(device);
```

The accessor `GetDeviceContext(handle)` is generated by the macro and is a constant-time pointer add — no hashtable lookup.

### FDO, PDO, filter — picking your role

| Role | Description | When |
|------|-------------|------|
| **FDO** (Function Device Object) | Owns the device. Default for `EvtDriverDeviceAdd`. | You wrote the bus-specific driver for the hardware. |
| **PDO** (Physical Device Object) | Created by the bus driver to represent a child enumerator. Use `WdfPdoInitAllocate`. | You're a bus driver enumerating children (e.g. virtual devices). |
| **Filter** | Sits above or below the FDO. `WdfFdoInitSetFilter(DeviceInit)` before `WdfDeviceCreate`. | You augment behaviour without owning the device. |

A filter MUST forward unhandled IRPs (`WdfRequestForwardToIoQueue` to a `Manual` queue, then `WdfRequestSend` to the I/O target). KMDF does this for you if you don't register a callback for that request type.

## Common pitfalls

| Pitfall | Why | Fix |
|---------|-----|-----|
| Storing kernel pointers in pageable context | Context is paged if you set `WDF_EXECUTION_LEVEL_PASSIVE` in some configs | Use `NonPagedPoolNx` for any data touched at DISPATCH |
| Calling `WdfObjectDelete` on a child | Parent owns it; double-free | Let parent destruction cascade, or reparent first |
| Confusing FDO with the underlying `DEVICE_OBJECT` | `WdfDeviceWdmGetDeviceObject` returns the WDM `DEVICE_OBJECT`, not the FDO concept | Use `WDFDEVICE` everywhere; only escape to WDM for legacy APIs |
| Filter that completes every IRP | Stalls the device stack | Forward unknown IRPs to the next driver via I/O target |
