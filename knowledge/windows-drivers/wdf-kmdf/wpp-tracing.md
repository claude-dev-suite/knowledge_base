# Windows Drivers - KMDF - WPP Software Tracing

> **Source**: https://learn.microsoft.com/en-us/windows-hardware/drivers/devtest/wpp-software-tracing
> **Skill**: dev-suite skill `windows/wdf-kmdf` — see SKILL.md for the always-loaded quick reference.

## What this covers

The end-to-end mechanics of Windows software Performance Trace (WPP) — preprocessor stage, control GUIDs, trace-message format strings, `.tmh` generation, runtime control with `tracelog`/`logman`, and decoding `.etl` to readable text using the driver's `.pdb`. Read when traces are missing, decoding fails, or you need a custom message function.

## Deep dive

### What WPP actually generates

WPP is a preprocessor (`tracewpp.exe`) that scans your sources for special macros (`TraceEvents`, `DoTraceMessage`) and emits a `.tmh` file per source. The `.tmh` contains:

- Per-message tracking structures with a unique ID
- A type-format table (so the decoder can format `%!STATUS!` etc.)
- Inlines that fan out into `EtwWriteTransfer` calls

The format strings themselves DO NOT ship in the binary — they live in the `.pdb`. The decoder reads them at decode time.

### Setting up control GUIDs

```c
// Trace.h
#define WPP_CONTROL_GUIDS                                              \
    WPP_DEFINE_CONTROL_GUID(                                           \
        MyDriverCtl, (00000000,1111,2222,3333,444444444444),           \
        WPP_DEFINE_BIT(TRACE_DRIVER)                                   \
        WPP_DEFINE_BIT(TRACE_DEVICE)                                   \
        WPP_DEFINE_BIT(TRACE_QUEUE)                                    \
        WPP_DEFINE_BIT(TRACE_PNP)                                      \
    )

#define WPP_LEVEL_FLAGS_LOGGER(lvl, flags) WPP_LEVEL_LOGGER(flags)
#define WPP_LEVEL_FLAGS_ENABLED(lvl, flags) \
    (WPP_LEVEL_ENABLED(flags) && WPP_CONTROL(WPP_BIT_##flags).Level >= lvl)
```

The macros at the bottom let you write `TraceEvents(LEVEL, FLAGS, "...")`. Without those, the default form is `TraceEvents(LEVEL, "...")`.

### Wiring into the build

In your `.vcxproj`:

```xml
<ItemDefinitionGroup>
  <WppTrace>
    <Enabled>true</Enabled>
    <ScanConfigurationData>Trace.h</ScanConfigurationData>
    <FunctionGenerationOptions>USEPREFIX(TraceEvents,LEVEL=,FLAGS=);</FunctionGenerationOptions>
  </WppTrace>
</ItemDefinitionGroup>
```

In each `.c` file:

```c
#include "Driver.h"
#include "Driver.tmh"     // generated by WPP — must be the LAST include
```

Include the `.tmh` AFTER the `WPP_CONTROL_GUIDS` definition is visible. If you forget, you get cryptic `WPP_BIT_xxx undeclared` errors.

### Initializing and tearing down

```c
NTSTATUS DriverEntry(PDRIVER_OBJECT DriverObject, PUNICODE_STRING RegistryPath) {
    WPP_INIT_TRACING(DriverObject, RegistryPath);
    // ... rest of DriverEntry; on failure call WPP_CLEANUP(DriverObject)
}

VOID MyEvtDriverUnload(WDFDRIVER Driver) {
    WPP_CLEANUP(WdfDriverWdmGetDriverObject(Driver));
}
```

`WPP_INIT_TRACING` registers the GUID with ETW and is required before any trace fires.

### Emitting traces

```c
TraceEvents(TRACE_LEVEL_INFORMATION, TRACE_DRIVER,
            "DriverEntry succeeded; status=%!STATUS!", status);

TraceEvents(TRACE_LEVEL_ERROR, TRACE_QUEUE,
            "Request %p failed with %!STATUS!", Request, status);

TraceEvents(TRACE_LEVEL_VERBOSE, TRACE_PNP,
            "EvtDeviceD0Entry: prev=%!WDF_POWER_DEVICE_STATE!", PrevState);
```

`%!FORMAT!` tokens are WPP type formatters. Common ones: `%!STATUS!`, `%!HRESULT!`, `%!GUID!`, `%!TIME!`, `%!PWSTR!`, `%!INSPECT!` (any object), `%!IRQL!`. You can register custom ones via `tracewpp -dll:myformatters.dll`.

### Capturing at runtime

```cmd
:: Start a session
tracelog -start MyTrace -guid #00000000-1111-2222-3333-444444444444 ^
         -f C:\trace.etl -flag 0xFF -level 5

:: Reproduce the issue, then stop
tracelog -stop MyTrace
```

`-flag` is a bitmask matching your `WPP_DEFINE_BIT` order; `-level` is `TRACE_LEVEL_*` (1=critical, 5=verbose). Use `0xFFFFFFFF` for everything.

### Decoding

```cmd
tracerpt C:\trace.etl -of CSV -o C:\trace.csv
:: Or use TraceView for live decoding:
TraceView -open Driver.pdb -guid #00000000-1111-2222-3333-444444444444
```

`tracerpt` needs symbols on `_NT_SYMBOL_PATH` to resolve `.pdb`s; otherwise rows say `<TraceFormatNotFound>`. You can also use `tracefmt` directly:

```cmd
tracefmt C:\trace.etl -p C:\symbols\Driver.pdb -o C:\trace.txt
```

### Custom message functions

Sometimes you want a wrapper that adds a prefix automatically:

```c
// Trace.h
#define WPP_LEVEL_DEVICE_FLAGS_LOGGER(lvl, dev, flags) WPP_LEVEL_LOGGER(flags)
#define WPP_LEVEL_DEVICE_FLAGS_ENABLED(lvl, dev, flags) \
    (WPP_LEVEL_ENABLED(flags) && WPP_CONTROL(WPP_BIT_##flags).Level >= lvl)
// Custom function macro
// FUNC TraceDev(LEVEL, DEVICE, FLAGS, MSG, ...);
// USESUFFIX(TraceDev, ", device=%p", DEVICE);
```

Now `TraceDev(TRACE_LEVEL_INFO, dev, TRACE_DRIVER, "ready")` automatically appends `device=...` in every emitted message.

### IFR — In-Flight Recorder

KMDF maintains a circular buffer of the last few hundred trace messages even without an active ETW session. Configure size via the `LogPages` registry value under your service. `!wdfkd.wdflogdump <DriverName>` in WinDbg dumps the IFR after a bugcheck — invaluable for diagnosing rare failures with no live trace running.

## Common pitfalls

| Pitfall | Why | Fix |
|---------|-----|-----|
| Including `.tmh` before `WPP_CONTROL_GUIDS` | Compile error | Move include after the GUID block |
| Stripped `.pdb` shipped to test | Decode shows `<TraceFormatNotFound>` | Keep matched `.pdb` for every released `.sys` |
| Calling `TraceEvents` before `WPP_INIT_TRACING` | First few messages dropped | Init WPP as the very first action in `DriverEntry` |
| Using `printf`-style `%s` for wide strings | Garbled text | Use `%!PWSTR!` or `%ls` |
| Re-using a GUID across drivers | Sessions collide | One GUID per driver, generated by `uuidgen` |
