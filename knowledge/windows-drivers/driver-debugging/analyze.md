# Windows Drivers - Driver Debugging - !analyze -v Anatomy

> **Source**: https://learn.microsoft.com/en-us/windows-hardware/drivers/debuggercmds/-analyze
> **Skill**: dev-suite skill `windows/driver-debugging` — see SKILL.md for the always-loaded quick reference.

## What this covers

Reading the output of `!analyze -v` line by line, the meaning of `BUCKET_ID`, `FAILURE_BUCKET_ID`, `FOLLOWUP_NAME`, when `IMAGE_NAME` is wrong, and how to override the default heuristic with `-show`, `-hang`, `-f`. Use when SKILL.md's "trust but verify" guidance is not enough.

### Key fields decoded

| Field | Meaning | When it lies |
|-------|---------|--------------|
| `BUGCHECK_CODE` | The kernel bugcheck number (e.g. `0xD1`) | Never |
| `BUGCHECK_Pn` | The four bugcheck arguments — meaning depends on the code | Always check the code's reference doc |
| `STACK_TEXT` | The faulting thread's call stack | When symbols are stale or `kP` shows different parameters |
| `MODULE_NAME` / `IMAGE_NAME` | Module that the heuristic blames | Often blames `nt` or `Wdf01000` when an upper driver is at fault — check `kP` and `!irp` |
| `FAILURE_BUCKET_ID` | Microsoft Online Crash Analysis bucket — useful for searching internal databases | Same as above |
| `BUCKET_ID` | Local de-dup key | Same |
| `FOLLOWUP_NAME` | Owner suggestion (e.g. `WDF`, `WIN32K`, `MachineOwner`) | "MachineOwner" = the driver itself |
| `PROCESS_NAME` | Process running on the faulting CPU at the time | Often `System` or `Idle` for kernel paths — not necessarily the cause |

### Useful options

```
!analyze -v                ; verbose (default for triage)
!analyze -v -hang          ; assume a hang (no bugcheck), find blocked threads
!analyze -show <code>      ; explain a bugcheck without running heuristics
!analyze -f                ; force re-run (after symbol reload)
!analyze -v -f -p          ; include performance / live data
!analyze -v -hang -p       ; live system hang triage
```

### Re-run flow when the first answer is wrong

```
.sympath+ srv*c:\sym*https://msdl.microsoft.com/download/symbols
.sympath+ c:\build\out\symbols      ; your driver's pdb
.reload /f
!analyze -v -f
kP                                   ; full param stack — verify FAULTING_IP matches
.trap <addr>                         ; switch to the trap frame
kP                                   ; the *real* faulting stack
!irp <irp_addr>                      ; if the bugcheck mentions an IRP
```

### When `IMAGE_NAME` blames the wrong driver

The heuristic walks the stack looking for the topmost non-Microsoft driver. It can be fooled when:

- A Microsoft driver (e.g. `Wdf01000`) calls back into your code via a callback pointer — the upper frames are MS, the *bug* is in your callback below
- Your driver tail-calls into another module (the return address belongs to the callee)
- The crash is in inline code generated by a compiler intrinsic

In all three cases: open `.trap`, run `kP`, and trust the actual stack over `IMAGE_NAME`.

## Common pitfalls

| Pitfall | Why | Fix |
|---------|-----|-----|
| Pasting `FAILURE_BUCKET_ID` into a bug report as the root cause | It's a search key, not a diagnosis | Include `kP`, `!irp`, and the real faulting source line |
| Skipping `.reload /f` before `!analyze` | Stale symbols → wrong line numbers | Always reload after the symbol path changes |
| Assuming `MachineOwner` means "your driver" | It means "not Microsoft" — could be any third-party driver loaded | Check `lm` for the actual module list |
| Using `!analyze` on a hang dump without `-hang` | Heuristic looks for a bugcheck that isn't there | Use `-hang -p` for live hangs |
| Ignoring `KEY_VALUES_STRING` | Microsoft's online analysis uses these tags to suggest known fixes | Search the bucket id on `learn.microsoft.com` |
