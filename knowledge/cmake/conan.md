# CMake - Conan 2 Integration

> **Source**: https://docs.conan.io/2/integrations/cmake.html
> **Skill**: dev-suite skill `cmake` — see SKILL.md for the always-loaded quick reference.

## What this covers

Conan 2's CMake integration via `CMakeToolchain` and `CMakeDeps`, the difference between `conanfile.txt` (declarative) and `conanfile.py` (imperative with full build logic), profiles for cross-compilation and ABI selection, and how Conan compares to vcpkg.

## Deep dive

### Conan 2 vs Conan 1

Conan 1 is end-of-life; Conan 2 is the current line (released 2023). The integration model differs:

- Conan 1: `cmake_find_package` and `cmake_paths` generators.
- Conan 2: `CMakeToolchain` + `CMakeDeps` generators producing a toolchain file and per-package config files.

Don't follow Conan 1 tutorials — the syntax has changed.

### conanfile.txt — declarative

```ini
[requires]
fmt/10.2.1
spdlog/1.14.1
boost/1.85.0

[generators]
CMakeToolchain
CMakeDeps

[options]
boost/*:shared=False
boost/*:without_test=True

[layout]
cmake_layout
```

The `cmake_layout` matches CMake's typical `build/<config>` layout. Conan writes generated files into `build/<config>/generators/`.

### conanfile.py — imperative

```python
from conan import ConanFile
from conan.tools.cmake import CMakeToolchain, CMakeDeps, CMake, cmake_layout

class MyApp(ConanFile):
    name = "myapp"
    version = "1.0.0"
    settings = "os", "compiler", "build_type", "arch"
    options = {"shared": [True, False], "with_zlib": [True, False]}
    default_options = {"shared": False, "with_zlib": True}

    def requirements(self):
        self.requires("fmt/10.2.1")
        self.requires("spdlog/1.14.1")
        if self.options.with_zlib:
            self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.tool_requires("cmake/3.28.0")
        self.test_requires("gtest/1.15.0")

    def layout(self):
        cmake_layout(self)

    def generate(self):
        tc = CMakeToolchain(self)
        tc.variables["MYAPP_USE_ZLIB"] = self.options.with_zlib
        tc.generate()
        deps = CMakeDeps(self)
        deps.generate()

    def build(self):
        cmake = CMake(self)
        cmake.configure()
        cmake.build()
```

Use `conanfile.py` when you need conditional dependencies, custom build steps, or to publish your own package to a Conan repo.

### Workflow

```bash
# 1. Install dependencies (downloads or builds them)
conan install . --build=missing -s build_type=Release

# 2. Configure CMake using the generated toolchain
cmake --preset conan-release        # preset auto-generated by Conan
cmake --build build/Release

# Or manually:
cmake -S . -B build \
  -DCMAKE_TOOLCHAIN_FILE=build/Release/generators/conan_toolchain.cmake \
  -DCMAKE_BUILD_TYPE=Release
cmake --build build
```

`cmake_layout` writes `build/Release/generators/conan_toolchain.cmake` and a matching `CMakeUserPresets.json` snippet — `cmake --preset conan-release` Just Works.

### Your CMakeLists.txt is unchanged

```cmake
find_package(fmt CONFIG REQUIRED)
find_package(spdlog CONFIG REQUIRED)
find_package(Boost REQUIRED COMPONENTS system)
target_link_libraries(myapp PRIVATE fmt::fmt spdlog::spdlog Boost::system)
```

`CMakeDeps` writes `<Pkg>Config.cmake` files that `find_package(... CONFIG)` finds via the toolchain's `CMAKE_PREFIX_PATH`.

### Profiles: ABI and cross-compile

A profile is a snapshot of build settings (compiler, flags, libcxx, build_type). Conan ships `default` and you can layer more:

```ini
# ~/.conan2/profiles/clang-asan
include(default)
[settings]
compiler=clang
compiler.version=17
compiler.libcxx=libc++
[conf]
tools.build:cxxflags=["-fsanitize=address,undefined"]
tools.build:exelinkflags=["-fsanitize=address,undefined"]
```

```bash
conan install . -pr:b default -pr:h clang-asan --build=missing
```

`-pr:b` is the *build* profile (the host running Conan), `-pr:h` is the *host* profile (the target binaries run on). For cross-compiling, set them differently:

```bash
conan install . -pr:b default -pr:h android-arm64 --build=missing
```

### Conan vs vcpkg — quick comparison

| Aspect | Conan 2 | vcpkg manifest |
|--------|---------|----------------|
| Config | `conanfile.{txt,py}` | `vcpkg.json` |
| Versioning | Per-package; revisions; lockfiles | Baseline pin |
| Binary cache | First-class (Artifactory, Conan Center) | Built-in (NuGet, files, S3) |
| Custom packages | `conan create` + push to private repo | Custom registry / overlay ports |
| Cross-compile | Profiles | Triplets |
| Imperative logic | `conanfile.py` | Limited (port portfile.cmake) |

Both work fine with modern CMake. Choose based on team familiarity, registry hosting, and ecosystem fit.

### CMakePresets integration

```json
{
  "include": ["build/Release/generators/CMakePresets.json"],
  "configurePresets": [
    {
      "name": "release",
      "inherits": "conan-release",
      "binaryDir": "${sourceDir}/build/Release"
    }
  ]
}
```

`include` (CMake 3.23+) lets you compose Conan's generated presets with your own without copy-pasting.

## Common pitfalls

| Pitfall | Why | Fix |
|---------|-----|-----|
| Following Conan 1 tutorials | API changed substantially | Use `docs.conan.io/2` exclusively |
| Forgetting `--build=missing` | Conan refuses to build deps not in cache | Always include for first install or new profiles |
| `find_package(...)` without `CONFIG` | May find a system module instead | Be explicit |
| Mixing build types in one folder | `cmake_layout` separates them per build_type | Use `cmake_layout`, accept the per-config dir |
| Editing generated `conan_toolchain.cmake` | Overwritten next install | Set variables in the profile or `tc.variables[...]` in `generate()` |
| Pushing private packages to public Conan Center | Accidental leak | Use a private remote (Artifactory, JFrog, self-hosted) |
