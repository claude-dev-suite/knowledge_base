# k6 Load Testing

## Overview

k6 is a modern load testing tool built by Grafana Labs. Tests are written in JavaScript (ES6), executed by a Go runtime for high performance, and produce structured metrics. k6 supports HTTP, WebSocket, gRPC, and browser protocols. It is designed for developer workflows and CI/CD integration.

## Setup

```bash
# macOS
brew install k6

# Windows
choco install k6

# Docker
docker run --rm -i grafana/k6 run - < test.js

# Verify installation
k6 version
```

## Test Script Structure

```javascript
// test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

// Options define load profile and thresholds
export const options = {
  vus: 10,               // Virtual users
  duration: '30s',       // Test duration
  thresholds: {
    http_req_duration: ['p(95)<500'],  // 95th percentile < 500ms
    http_req_failed: ['rate<0.01'],    // Error rate < 1%
  },
};

// Setup (runs once before VUs start)
export function setup() {
  const loginRes = http.post('https://api.example.com/auth/login', JSON.stringify({
    username: 'loadtest',
    password: 'password',
  }), { headers: { 'Content-Type': 'application/json' } });

  return { token: loginRes.json('token') };
}

// Default function (executed by each VU in a loop)
export default function (data) {
  const headers = {
    Authorization: `Bearer ${data.token}`,
    'Content-Type': 'application/json',
  };

  // GET request
  const productsRes = http.get('https://api.example.com/products', { headers });

  check(productsRes, {
    'status is 200': (r) => r.status === 200,
    'response has products': (r) => r.json('products').length > 0,
    'response time < 500ms': (r) => r.timings.duration < 500,
  });

  sleep(1); // Think time between iterations
}

// Teardown (runs once after all VUs finish)
export function teardown(data) {
  http.post('https://api.example.com/auth/logout', null, {
    headers: { Authorization: `Bearer ${data.token}` },
  });
}
```

## VUs and Iterations

**Virtual Users (VUs)** execute the default function in a loop. Each VU is an independent execution context with its own cookies, connections, and variables.

**Iterations** are complete executions of the default function. Each VU performs multiple iterations over the test duration.

```javascript
export const options = {
  // Fixed VUs for duration
  vus: 50,
  duration: '5m',

  // OR fixed iterations total
  iterations: 1000,
  vus: 10,  // 10 VUs share 1000 iterations
};
```

## HTTP Requests

```javascript
import http from 'k6/http';

export default function () {
  // GET
  const getRes = http.get('https://api.example.com/products/123');

  // POST with JSON body
  const postRes = http.post(
    'https://api.example.com/orders',
    JSON.stringify({
      productId: '123',
      quantity: 2,
    }),
    {
      headers: { 'Content-Type': 'application/json' },
      tags: { name: 'CreateOrder' },  // Custom tag for grouping metrics
    }
  );

  // PUT
  http.put('https://api.example.com/products/123',
    JSON.stringify({ price: 19.99 }),
    { headers: { 'Content-Type': 'application/json' } }
  );

  // DELETE
  http.del('https://api.example.com/products/123');

  // Batch requests (parallel)
  const responses = http.batch([
    ['GET', 'https://api.example.com/products'],
    ['GET', 'https://api.example.com/categories'],
    ['GET', 'https://api.example.com/featured'],
  ]);
}
```

## Checks and Thresholds

### Checks (Assertions)

```javascript
import { check } from 'k6';

check(response, {
  'status is 200': (r) => r.status === 200,
  'body contains expected field': (r) => r.json('id') !== undefined,
  'response time acceptable': (r) => r.timings.duration < 500,
  'content type is JSON': (r) => r.headers['Content-Type'].includes('application/json'),
  'body size is reasonable': (r) => r.body.length < 10000,
});
```

### Thresholds (Pass/Fail Criteria)

```javascript
export const options = {
  thresholds: {
    // Built-in metrics
    http_req_duration: ['p(95)<500', 'p(99)<1000', 'avg<200'],
    http_req_failed: ['rate<0.01'],
    http_reqs: ['rate>100'],          // Minimum throughput
    checks: ['rate>0.99'],            // 99% of checks must pass

    // Tagged metrics
    'http_req_duration{name:CreateOrder}': ['p(95)<800'],
    'http_req_duration{name:GetProduct}': ['p(95)<200'],

    // Custom metrics
    'order_processing_time': ['p(95)<1000'],
  },
};
```

## Scenarios

Scenarios provide fine-grained control over VU scheduling and load patterns.

### Constant VUs

```javascript
export const options = {
  scenarios: {
    constant_load: {
      executor: 'constant-vus',
      vus: 50,
      duration: '5m',
    },
  },
};
```

### Ramping VUs

```javascript
export const options = {
  scenarios: {
    ramp_up_down: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '2m', target: 50 },   // Ramp up to 50
        { duration: '5m', target: 50 },   // Stay at 50
        { duration: '2m', target: 100 },  // Ramp to 100
        { duration: '5m', target: 100 },  // Stay at 100
        { duration: '2m', target: 0 },    // Ramp down
      ],
      gracefulRampDown: '30s',
    },
  },
};
```

### Constant Arrival Rate

Controls the request rate regardless of response time. Adds VUs as needed to maintain the rate.

```javascript
export const options = {
  scenarios: {
    constant_rate: {
      executor: 'constant-arrival-rate',
      rate: 100,              // 100 iterations per timeUnit
      timeUnit: '1s',         // Per second
      duration: '5m',
      preAllocatedVUs: 50,    // Initial VU pool
      maxVUs: 200,            // Max VUs if needed
    },
  },
};
```

### Ramping Arrival Rate

```javascript
export const options = {
  scenarios: {
    ramping_rate: {
      executor: 'ramping-arrival-rate',
      startRate: 10,
      timeUnit: '1s',
      preAllocatedVUs: 50,
      maxVUs: 300,
      stages: [
        { duration: '2m', target: 50 },
        { duration: '5m', target: 50 },
        { duration: '2m', target: 100 },
        { duration: '5m', target: 100 },
        { duration: '2m', target: 0 },
      ],
    },
  },
};
```

### Multiple Scenarios

```javascript
export const options = {
  scenarios: {
    browse: {
      executor: 'constant-vus',
      vus: 30,
      duration: '10m',
      exec: 'browseProducts',
    },
    purchase: {
      executor: 'constant-arrival-rate',
      rate: 5,
      timeUnit: '1s',
      duration: '10m',
      preAllocatedVUs: 10,
      maxVUs: 50,
      exec: 'purchaseFlow',
    },
  },
};

export function browseProducts() {
  http.get('https://api.example.com/products');
  sleep(2);
}

export function purchaseFlow() {
  http.post('https://api.example.com/orders', JSON.stringify({ productId: '123' }), {
    headers: { 'Content-Type': 'application/json' },
  });
  sleep(1);
}
```

## Custom Metrics

```javascript
import { Counter, Gauge, Rate, Trend } from 'k6/metrics';

const orderCount = new Counter('orders_created');
const activeUsers = new Gauge('active_users');
const successRate = new Rate('order_success_rate');
const orderDuration = new Trend('order_processing_time');

export default function () {
  const start = Date.now();

  const res = http.post('https://api.example.com/orders',
    JSON.stringify({ productId: '123', quantity: 1 }),
    { headers: { 'Content-Type': 'application/json' } }
  );

  const duration = Date.now() - start;

  orderCount.add(1);
  activeUsers.add(1);
  successRate.add(res.status === 201);
  orderDuration.add(duration);
}
```

## CI/CD Integration

### GitHub Actions

```yaml
name: Load Test
on:
  pull_request:
    branches: [main]

jobs:
  load-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install k6
        run: |
          sudo gpg -k
          sudo gpg --no-default-keyring --keyring /usr/share/keyrings/k6-archive-keyring.gpg --keyserver hkp://keyserver.ubuntu.com:80 --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D68
          echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" | sudo tee /etc/apt/sources.list.d/k6.list
          sudo apt-get update && sudo apt-get install k6

      - name: Run smoke test
        run: k6 run --out json=results.json tests/load/smoke.js

      - name: Upload results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: k6-results
          path: results.json
```

### Docker in CI

```yaml
- name: Run load test
  run: |
    docker run --rm -v $(pwd)/tests:/tests \
      -e BASE_URL=${{ secrets.STAGING_URL }} \
      grafana/k6 run /tests/load/smoke.js
```

## Anti-Patterns

1. **No think time** - Real users pause between actions. Without `sleep()`, you generate unrealistic thundering-herd traffic.
2. **Testing from a single region** - Load should originate from where users are. Use distributed execution for realistic latency.
3. **Ignoring ramping** - Jumping to full load instantly stresses autoscaling and warm-up. Always ramp up.
4. **Hardcoded test data** - Using the same product ID for every request hits cache too often. Use parameterized data.
5. **No thresholds** - Without pass/fail criteria, the test is informational only. Set thresholds for CI gates.
6. **Testing only happy path** - Include error scenarios, 404s, and edge cases in your load test mix.

## Production Checklist

- [ ] Test scripts reviewed and version controlled alongside application code
- [ ] Thresholds defined based on SLOs (p95, p99, error rate)
- [ ] Think time and realistic user pacing included
- [ ] Test data parameterized to avoid cache bias
- [ ] Scenarios model real traffic patterns (browse, search, purchase)
- [ ] Results output to a persistent store (InfluxDB, Grafana Cloud)
- [ ] CI pipeline runs smoke tests on every PR
- [ ] Full load tests run on staging before releases
- [ ] Baseline results documented for regression comparison
- [ ] Infrastructure monitoring correlated with k6 metrics during tests
