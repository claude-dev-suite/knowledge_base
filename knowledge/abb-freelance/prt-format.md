## Overview

The `.prt` file is the central engineering database of an ABB Freelance project. Its full name is **Parameter Record Table**. It stores:
- Controller hardware configuration (modules, node addresses, fieldbus slave lists)
- The complete variable/tag list with point IDs, data types, and HMI-visibility flags
- Function block instance parameter values
- Task configuration (cyclic intervals, priorities, watchdog times)
- Alarm definitions and limits
- A project-level checksum for integrity validation

Control Builder F reads and writes `.prt` exclusively — it is not a format meant for direct hand-editing. However, understanding its structure is essential for:
- Generating large tag lists programmatically (mass engineering)
- Debugging CBF import failures
- Implementing custom Python/script-based bulk-engineering tools

## File Encoding

```
Encoding:  UTF-16LE with BOM (Byte Order Mark)
BOM bytes: FF FE  (at file start, before any text)
Line ending: CRLF (\r\n)
Field separator: TAB (\t) within records
```

**Critical:** A `.prt` file opened in a text editor that does not respect UTF-16LE will appear as corrupted binary (every ASCII character padded with a null byte). Python's `open()` with `encoding='utf-16'` (note: Python auto-detects LE/BE from BOM when `'utf-16'` is specified, or use `'utf-16-le'` and manually handle the BOM) correctly reads the file.

**Python read pattern:**
```python
with open('project.prt', encoding='utf-16', errors='replace') as f:
    lines = f.readlines()
```

**Python write pattern (must produce UTF-16LE with BOM):**
```python
import codecs
with codecs.open('project.prt', 'w', encoding='utf-16-le') as f:
    f.write('\ufeff')  # Write BOM manually when using utf-16-le
    for line in records:
        f.write(line + '\r\n')
```
Or use `encoding='utf-16'` which writes the BOM automatically.

## DUMP_FILETYPE Values

The first non-empty line of a `.prt` file after the BOM is the `DUMP_FILETYPE` record, which identifies what kind of export this file represents.

| Value | Meaning |
|-------|---------|
| `DUMP_FILETYPE\t101` | Full project export — complete controller configuration including all sections |
| `DUMP_FILETYPE\t102` | Partial project export — typically a variable list or parameter subset for import/merge into an existing project |

Type 102 files are generated by CBF's **Bulk Data Manager** export and are the format used for `.eam`-style imports. Type 101 is the full `.prt` database written when CBF saves the project.

## File Structure — Sections

A `.prt` file is organized in named sections. Each section begins with a section header line and ends with either the next section header or the end-of-file. Within a section, each record is one tab-delimited line.

### High-Level Structure

```
DUMP_FILETYPE\t101\r\n
VERSION\t<version_string>\r\n
PROJECTNAME\t<project_name>\r\n
\r\n
[NODE section]
\r\n
[DBS section]
\r\n
[EAM section]
\r\n
[MSR section]
\r\n
[LAD section]
\r\n
[PARA section]
\r\n
[CHECKSUM section]
```

Empty lines between sections are separators, not content.

---

### Section: NODE

**Purpose:** Defines the controllers (process stations / nodes) in the project and their hardware configuration.

**Header line:**
```
NODE\tCOUNT\t<n>
```
Where `<n>` = number of node records that follow.

**Node record format:**
```
NODE\t<NodeID>\t<NodeName>\t<NodeType>\t<IPAddress>\t<Description>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT (1–254) | `5` | Logical DIGI node identifier |
| `NodeName` | STRING | `AC800F_UNIT1` | Symbolic name, max 24 chars, ASCII only |
| `NodeType` | STRING | `AC800F` | Controller model: `AC700F`, `AC800F`, `AC900F` |
| `IPAddress` | STRING | `192.168.1.5` | Controller Ethernet IP |
| `Description` | STRING | `Primary reactor controller` | Free text; may be empty |

**Full NODE section example:**
```
NODE	COUNT	3
NODE	1	AC800F_UTIL	AC800F	192.168.1.1	Utilities
NODE	5	AC800F_REACT	AC800F	192.168.1.5	Reactor section
NODE	9	AC900F_MAIN	AC900F	192.168.1.9	Main process
```

**Corner cases:**
- NodeID 0 is reserved; avoid using it
- NodeName must be unique within the project
- If IPAddress is left empty, DIGI communication cannot be established at runtime but the project will compile
- NodeType string is case-sensitive; `ac800f` (lowercase) may cause CBF to reject the record on import

---

### Section: DBS (Database / Variable List)

**Purpose:** The full tag/variable list — every process variable in the project that has been assigned a point ID. This is the core of what the HMI binds to.

**Header line:**
```
DBS\tCOUNT\t<n>
```

**Variable record format:**
```
DBS\t<NodeID>\t<PointID>\t<SymbolicName>\t<DataType>\t<HMIFlag>\t<Description>\t<Units>\t<InitValue>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT | `5` | References a NODE record |
| `PointID` | INT (1–65535) | `1042` | Unique per NodeID; stable identifier |
| `SymbolicName` | STRING | `P_101_RUN` | IEC variable name; ASCII, max 32 chars |
| `DataType` | STRING | `BOOL` | IEC type: `BOOL`, `INT`, `DINT`, `REAL`, `TIME`, `STRING` |
| `HMIFlag` | INT | `1` | 0 = internal only; 1 = visible to DigiVis/OPC |
| `Description` | STRING | `Pump 101 running status` | Shown in DigiVis tooltip and alarm list |
| `Units` | STRING | `` | Engineering units string (blank for BOOL) |
| `InitValue` | STRING | `FALSE` | Initial value on power-up/download |

**Full DBS section example:**
```
DBS	COUNT	6
DBS	5	1001	P_101_CMD_ON	BOOL	1	Pump 101 start command
DBS	5	1002	P_101_CMD_OFF	BOOL	1	Pump 101 stop command
DBS	5	1003	P_101_RUN	BOOL	1	Pump 101 run output		FALSE
DBS	5	1004	P_101_FLR	BOOL	1	Pump 101 fault alarm		FALSE
DBS	5	1005	P_101_ILK	BOOL	1	Pump 101 interlock alarm	FALSE
DBS	5	1042	FT_201_OUT	REAL	1	Flow transmitter 201	m3/h	0.0
```

**Corner cases:**
- `PointID` values must be globally unique **per NodeID**, not globally unique across the project
- Two variables on different nodes can share the same PointID — this is valid and common
- `PointID` 0 is reserved; do not use
- `DataType` mismatch between DBS record and the actual IEC variable type causes a compile warning but not always an error; the runtime may silently truncate or misinterpret values
- `Description` field containing a TAB character will corrupt the record's column alignment; CBF strips tabs on export but third-party generators must escape them
- `HMIFlag = 0` variables are still accessible from Control Builder F for debugging but do not appear in DigiVis tag browser or OPC namespace
- Bulk-generated DBS records must have PointIDs allocated in non-colliding ranges; no auto-allocation mechanism exists outside CBF's UI

---

### Section: EAM (Export/Import variable Assignment Map)

**Purpose:** A mapping layer that associates symbolic variable names with their DIGI HMI display properties: alarm classes, process value ranges for display scaling, and alarm priority levels.

**Header line:**
```
EAM\tCOUNT\t<n>
```

**EAM record format:**
```
EAM\t<NodeID>\t<PointID>\t<AlarmClass>\t<LoRange>\t<HiRange>\t<Priority>\t<AckRequired>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT | `5` | Must match a DBS record |
| `PointID` | INT | `1042` | Must match a DBS record |
| `AlarmClass` | STRING | `PROCESS` | Alarm class name; defined in project alarm config |
| `LoRange` | REAL | `0.0` | Display range low (for analog tags — used by DigiVis bar graph) |
| `HiRange` | REAL | `100.0` | Display range high |
| `Priority` | INT | `2` | Alarm priority 1–5 (1 = highest) |
| `AckRequired` | INT | `1` | 1 = operator acknowledgement required; 0 = auto-ack |

**EAM section example:**
```
EAM	COUNT	2
EAM	5	1004	PROCESS	0.0	1.0	2	1
EAM	5	1042	PROCESS	0.0	500.0	3	1
```

**Corner cases:**
- EAM records are optional for BOOL tags (they have no meaningful range)
- Missing EAM record for an analog tag causes DigiVis to display the bar graph full-scale (0–100%) regardless of actual value range
- `AlarmClass` must exactly match one of the alarm classes defined in the project; a mismatch causes CBF to silently assign the default alarm class

---

### Section: MSR (Measurement / Analog Tag Details)

**Purpose:** Extended properties for analog (REAL) process variables — engineering unit labels, format strings for display, and deadband for alarm evaluation.

**Header line:**
```
MSR\tCOUNT\t<n>
```

**MSR record format:**
```
MSR\t<NodeID>\t<PointID>\t<EULabel>\t<Format>\t<Deadband>\t<FilterTime>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT | `5` | |
| `PointID` | INT | `1042` | |
| `EULabel` | STRING | `m3/h` | Displayed as unit label in DigiVis faceplate |
| `Format` | STRING | `%.1f` | C-style printf format for numeric display |
| `Deadband` | REAL | `0.5` | Hysteresis for alarm transitions |
| `FilterTime` | INT | `0` | Additional DigiVis-side display filter in seconds (0 = no extra filter) |

**MSR section example:**
```
MSR	COUNT	1
MSR	5	1042	m3/h	%.1f	1.0	0
```

**Corner cases:**
- `Format` strings wider than the DigiVis faceplate display field get truncated; `%.3f` on a 6-digit value with units will overflow the display cell
- `EULabel` is independent of the `Units` field in the DBS record; they should match but CBF does not enforce consistency

---

### Section: LAD (Ladder / Logic Alarm Definitions)

**Purpose:** Defines discrete alarm point properties — the message text, alarm class, and acknowledgement behavior for BOOL alarm outputs from function blocks (e.g., `FLR`, `ILK`, `HH_ALM` from standard blocks).

**Header line:**
```
LAD\tCOUNT\t<n>
```

**LAD record format:**
```
LAD\t<NodeID>\t<PointID>\t<AlarmText>\t<AlarmClass>\t<Priority>\t<AckRequired>\t<Horn>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT | `5` | |
| `PointID` | INT | `1004` | Must be a BOOL variable in DBS |
| `AlarmText` | STRING | `Pump 101 FAULT` | Text shown in DigiVis alarm list |
| `AlarmClass` | STRING | `FAULT` | Must match defined alarm class |
| `Priority` | INT | `1` | 1 (highest) – 5 (lowest) |
| `AckRequired` | INT | `1` | |
| `Horn` | INT | `1` | 1 = activate audible alarm horn; 0 = silent |

**LAD section example:**
```
LAD	COUNT	3
LAD	5	1004	Pump 101 FAULT	FAULT	1	1	1
LAD	5	1005	Pump 101 INTERLOCK	INTERLOCK	2	1	0
LAD	5	1006	FT-201 HIGH FLOW	PROCESS	3	1	0
```

**Corner cases:**
- `AlarmText` containing special characters (double quotes, tabs) must be stripped; CBF escapes neither — a tab in AlarmText shifts all subsequent fields right by one column and corrupts the record
- LAD records for non-alarm BOOL variables (e.g., run outputs) are ignored but generate a CBF import warning

---

### Section: PARA (Function Block Parameters)

**Purpose:** Stores the configured parameter values for all function block instances in the project. This is what differentiates a template-generated project from a manually tuned one — all `START_T`, `STOP_T`, alarm limits, PID tuning constants, etc.

**Header line:**
```
PARA\tCOUNT\t<n>
```

**PARA record format:**
```
PARA\t<NodeID>\t<POU>\t<InstanceName>\t<ParamName>\t<Value>
```

| Field | Type | Example | Notes |
|-------|------|---------|-------|
| `NodeID` | INT | `5` | |
| `POU` | STRING | `Main` | Program/POU name where the FB instance lives |
| `InstanceName` | STRING | `P_101` | FB instance name in the POU |
| `ParamName` | STRING | `START_T` | Parameter input name |
| `Value` | STRING | `T#10s` | Value in IEC literal syntax |

**PARA section example:**
```
PARA	COUNT	8
PARA	5	Main	P_101	START_T	T#10s
PARA	5	Main	P_101	STOP_T	T#8s
PARA	5	Main	P_101	IC1	%IX0.0.1
PARA	5	Main	FT_201	EU_LO	0.0
PARA	5	Main	FT_201	EU_HI	500.0
PARA	5	Main	FT_201	HH_LIM	450.0
PARA	5	Main	FT_201	HI_LIM	400.0
PARA	5	Main	FT_201	FILTER_T	T#5s
```

**Corner cases:**
- `Value` must be in the correct IEC literal syntax for the parameter type:
  - BOOL: `TRUE` or `FALSE` (case-sensitive)
  - TIME: `T#10s`, `T#100ms`, `T#1m30s`
  - REAL: `0.0` (must include decimal point; `0` alone is interpreted as INT in some CBF versions)
  - INT/DINT: plain integer, no quotes
  - STRING: `'text value'` with single quotes
- I/O address literals (e.g., `%IX0.0.1`) reference the physical I/O binding and must exactly match the hardware config; a mismatch generates a compile error but not always an explicit message pointing to the PARA record
- PARA records for non-existent FB instances are silently skipped on import; a typo in `InstanceName` means the parameter is never applied

---

### Section: CHECKSUM

**Purpose:** A CRC/hash over the file content to detect corruption or unauthorized modification. CBF validates the checksum on project open.

**Format:**
```
CHECKSUM\t<algorithm>\t<value>
```

| Field | Example | Notes |
|-------|---------|-------|
| `algorithm` | `CRC32` | Checksum algorithm identifier |
| `value` | `A3F2C891` | Hex-encoded checksum value |

**Example:**
```
CHECKSUM	CRC32	A3F2C891
```

**Critical — Checksum Rules for Generated Files:**
When generating a `.prt` file programmatically, the checksum must be recalculated over the file content. The checksum covers the entire file content **excluding** the CHECKSUM line itself.

**Python checksum calculation pattern:**
```python
import binascii

def calculate_prt_checksum(content_lines: list[str]) -> str:
    """
    Calculate CRC32 checksum for PRT file content.
    content_lines: list of strings, each WITHOUT trailing CRLF.
    Returns hex string (uppercase, 8 chars).
    """
    # Join with CRLF, encode back to UTF-16LE (without BOM for checksum body)
    body = '\r\n'.join(content_lines)
    body_bytes = body.encode('utf-16-le')
    crc = binascii.crc32(body_bytes) & 0xFFFFFFFF
    return f'{crc:08X}'
```

**Corner case:** If the generated file uses `utf-16` (with BOM prepended) vs `utf-16-le` (no BOM), the checksum changes because the BOM bytes are included or excluded. ABB's CBF writes the file with BOM but the checksum calculation in some versions covers only the content after the BOM. Test with a known-good CBF-generated file before deploying a generator.

## Template Replacement Pattern for Motors

A common mass-engineering pattern: maintain a **motor template** PRT fragment and do string replacement for each motor tag.

### Template Fragment

```
# Motor template — replace MOTOR_TAG with actual tag, NODE_X with NodeID,
#                           PID_START with starting PointID

DBS	NODE_X	PID_START	MOTOR_TAG_CMD_ON	BOOL	1	MOTOR_TAG start command
DBS	NODE_X	PID_START+1	MOTOR_TAG_CMD_OFF	BOOL	1	MOTOR_TAG stop command
DBS	NODE_X	PID_START+2	MOTOR_TAG_RUN	BOOL	1	MOTOR_TAG running status		FALSE
DBS	NODE_X	PID_START+3	MOTOR_TAG_FLR	BOOL	1	MOTOR_TAG fault alarm		FALSE
DBS	NODE_X	PID_START+4	MOTOR_TAG_ILK	BOOL	1	MOTOR_TAG interlock alarm	FALSE
DBS	NODE_X	PID_START+5	MOTOR_TAG_STATE	INT	1	MOTOR_TAG state code		0

LAD	NODE_X	PID_START+3	MOTOR_TAG FAULT	FAULT	1	1	1
LAD	NODE_X	PID_START+4	MOTOR_TAG INTERLOCK	INTERLOCK	2	1	0

PARA	NODE_X	Main	MOTOR_TAG	START_T	T#10s
PARA	NODE_X	Main	MOTOR_TAG	STOP_T	T#8s
```

### Python Generation Pattern

```python
import codecs
import binascii

MOTOR_TEMPLATE = [
    # (section, fields_template)
    ('DBS', '{node}\t{pid}\t{tag}_CMD_ON\tBOOL\t1\t{tag} start command\t\t'),
    ('DBS', '{node}\t{pid1}\t{tag}_CMD_OFF\tBOOL\t1\t{tag} stop command\t\t'),
    ('DBS', '{node}\t{pid2}\t{tag}_RUN\tBOOL\t1\t{tag} running\t\tFALSE'),
    ('DBS', '{node}\t{pid3}\t{tag}_FLR\tBOOL\t1\t{tag} fault\t\tFALSE'),
    ('DBS', '{node}\t{pid4}\t{tag}_ILK\tBOOL\t1\t{tag} interlock\t\tFALSE'),
    ('DBS', '{node}\t{pid5}\t{tag}_STATE\tINT\t1\t{tag} state\t\t0'),
    ('LAD', '{node}\t{pid3}\t{tag} FAULT\tFAULT\t1\t1\t1'),
    ('LAD', '{node}\t{pid4}\t{tag} INTERLOCK\tINTERLOCK\t2\t1\t0'),
    ('PARA', '{node}\tMain\t{tag}\tSTART_T\tT#10s'),
    ('PARA', '{node}\tMain\t{tag}\tSTOP_T\tT#8s'),
]

def generate_motor_records(tag: str, node_id: int, base_pid: int) -> dict:
    """Returns dict of section -> list of record strings."""
    records = {}
    pids = [base_pid + i for i in range(6)]
    ctx = {
        'tag': tag, 'node': node_id,
        'pid': pids[0], 'pid1': pids[1], 'pid2': pids[2],
        'pid3': pids[3], 'pid4': pids[4], 'pid5': pids[5],
    }
    for section, tmpl in MOTOR_TEMPLATE:
        records.setdefault(section, []).append(
            f'{section}\t' + tmpl.format(**ctx)
        )
    return records

def write_prt(
    output_path: str,
    project_name: str,
    nodes: list[dict],
    motors: list[dict]
) -> None:
    """
    nodes = [{'id': 5, 'name': 'AC800F_REACT', 'type': 'AC800F', 'ip': '192.168.1.5', 'desc': ''}]
    motors = [{'tag': 'P_101', 'node': 5, 'base_pid': 1001}]
    """
    all_records = {'NODE': [], 'DBS': [], 'LAD': [], 'PARA': []}

    # NODE records
    for n in nodes:
        all_records['NODE'].append(
            f"NODE\t{n['id']}\t{n['name']}\t{n['type']}\t{n['ip']}\t{n.get('desc','')}"
        )

    # Motor records
    for m in motors:
        recs = generate_motor_records(m['tag'], m['node'], m['base_pid'])
        for section, lines in recs.items():
            all_records.setdefault(section, []).extend(lines)

    # Assemble file lines (without CHECKSUM yet)
    lines = [
        f'DUMP_FILETYPE\t101',
        f'VERSION\t9.2SP1',
        f'PROJECTNAME\t{project_name}',
        '',
        f'NODE\tCOUNT\t{len(all_records["NODE"])}',
        *all_records['NODE'],
        '',
        f'DBS\tCOUNT\t{len(all_records["DBS"])}',
        *all_records['DBS'],
        '',
        f'LAD\tCOUNT\t{len(all_records["LAD"])}',
        *all_records['LAD'],
        '',
        f'PARA\tCOUNT\t{len(all_records["PARA"])}',
        *all_records['PARA'],
    ]

    # Calculate checksum
    body = '\r\n'.join(lines)
    body_bytes = body.encode('utf-16-le')
    crc = binascii.crc32(body_bytes) & 0xFFFFFFFF
    checksum_line = f'CHECKSUM\tCRC32\t{crc:08X}'
    lines.append('')
    lines.append(checksum_line)

    # Write as UTF-16LE with BOM
    with codecs.open(output_path, 'w', encoding='utf-16-le') as f:
        f.write('\ufeff')  # BOM
        f.write('\r\n'.join(lines))

# Usage:
# write_prt(
#     'generated_project.prt',
#     'ReactorUnit1',
#     nodes=[{'id': 5, 'name': 'AC800F_REACT', 'type': 'AC800F', 'ip': '192.168.1.5', 'desc': 'Reactor'}],
#     motors=[
#         {'tag': 'P_101', 'node': 5, 'base_pid': 1001},
#         {'tag': 'P_102', 'node': 5, 'base_pid': 1007},
#         {'tag': 'P_103', 'node': 5, 'base_pid': 1013},
#     ]
# )
```

## Mandatory Fields and Defaults

Not all sections are required in every `.prt` file. A minimal valid DUMP_FILETYPE 102 (import fragment) can contain only `DBS` records. The complete mandatory/optional breakdown:

| Section | Required in Type 101 | Required in Type 102 | Default if absent |
|---------|---------------------|---------------------|-------------------|
| `NODE` | Yes | No | No nodes defined |
| `DBS` | Yes | Yes (at least 1) | Empty variable list |
| `EAM` | No | No | No alarm class / range |
| `MSR` | No | No | No EU label / format |
| `LAD` | No | No | No alarm text |
| `PARA` | No | No | FB defaults remain |
| `CHECKSUM` | Yes | Yes | CBF rejects on open |

## Corner Cases and Common Errors

1. **BOM on DUMP_FILETYPE 102 partial imports**: CBF expects the BOM to be present even on Type 102 imports. A file written without the BOM (e.g., UTF-8 or ASCII) will cause CBF to report "Invalid file format" rather than a more helpful message.

2. **Tab vs space confusion**: Fields are tab-separated. Any space used as a separator causes the field count to shift; CBF's parser is strict about the exact field positions. Use tab (`\t`, U+0009) exclusively.

3. **Trailing tab on records**: Some CBF versions write a trailing tab on records with optional empty trailing fields. A generator that omits the trailing tab may cause CBF to report "Too few fields in record" on import for those versions. Safe practice: always write the full expected number of tab-delimited fields even if trailing ones are empty.

4. **PointID allocation gaps**: PointIDs do not need to be contiguous. Gaps are valid and are the norm in large projects (motors allocated 1001–1006, 1007–1012, etc. regardless of insertions/deletions). Do not compact PointIDs to fill gaps — this invalidates DIGI addresses already stored in `.dmf` display files and OPC client configurations.

5. **PointID reuse after deletion**: Deleting a variable in CBF's UI frees its PointID. If the same PointID is assigned to a new variable, and there is a DigiVis display or external OPC client that cached the old binding, the old client will silently read the wrong variable. Safe practice: treat PointIDs as permanent — retire (mark as unused) rather than delete and reuse.

6. **REAL literal format**: IEC REAL literals must include a decimal point. The value `0` without a decimal in a PARA record targeting a REAL parameter will be interpreted as INT(0) in some CBF versions and cause a type mismatch compile warning; the value is still applied as 0.0 but the warning obscures other real errors. Always write `0.0`.

7. **TIME literal edge cases**: `T#0ms` is valid and means zero time (disable). `T#0s` is also valid. `T#0` (no unit suffix) is not valid and causes a parse error. Large time values: `T#99999s` is at the edge of the TIME type's representable range (depends on CBF version); safe maximum is `T#9999s` or use `T#2h46m39s`.

8. **Special characters in Description / AlarmText**: TAB (`\t`) corrupts column alignment. LF (`\n`) within a field (if somehow inserted) corrupts the line parsing. Single quotes (`'`) in Description fields for STRING parameters must be doubled: `''`. Double quotes (`"`) are not delimiters in `.prt` — they pass through literally.

9. **Version mismatch on import**: A `.prt` file with `VERSION\t9.2SP1` imported into CBF running Freelance 2019 will trigger a version upgrade warning. The import still succeeds but CBF may rewrite sections in the newer format, invalidating any downstream diff/comparison tool that tracked the original format.

10. **CHECKSUM failure on hand-edit**: Any manual text edit to a `.prt` file invalidates the checksum. CBF will refuse to open the file. After any programmatic modification of a `.prt`, the checksum section must be recalculated and rewritten before the file is loaded into CBF.
